# KB Authoring ‚Äî Fact Stability Tier Model

> Per-fact classification used inside KB documents. Orthogonal to the per-doc
> `kb-category:` field (which classifies whole documents).

Every claim a KB document makes belongs to one of four stability tiers. The tier
determines whether the claim is allowed inline, where it should live, and how the
reviewer treats it.

## The four tiers

### T1 ‚Äî Concept

Patterns, definitions, architectural laws, contracts, threat models.

- **Examples:**
  - "The Knowledge Base is the gravitational center; every phase reads it."
  - "Reviewers run with clean context ‚Äî they must never see the executor's working notes."
  - "Agents are organized into three tiers (Opus / Sonnet / Haiku) by precision needed."
- **Stability:** months. Changes only on real methodological revision.
- **Inline?** Yes ‚Äî this IS the knowledge.
- **Review treatment:** full review. The reviewer evaluates whether the claim is correct,
  complete, and consistent with the rest of the KB.

### T2 ‚Äî Structure

Cardinality contracts, schemas, fixed-shape declarations, named-list assertions.

- **Examples:**
  - "There are **16 standard PRIMARY KB documents**." (cardinality contract ‚Äî downstream
    skills depend on exactly this count; meta + extension + generated docs are separate
    and NOT part of the 16)
  - "Each task has exactly ONE type, drawn from a fixed list of 8."
  - "Every agent frontmatter must declare `name`, `description`, `model`, `tools`."
- **Stability:** weeks. Changes when methodology evolves a contract.
- **Inline?** Yes ‚Äî but ONLY when the count or shape is a contract (downstream code or
  skill depends on it). Pure "size" counts (line counts, byte counts) are NOT T2.
- **Review treatment:** full review against disk truth. Declared in `contracts:`
  frontmatter field; lint enforces. (See [frontmatter-schema.md](frontmatter-schema.md).)

### T3 ‚Äî Metric

Numeric measurements of disk state: line counts, byte counts, file sizes, function
counts, term counts when not a contract.

- **Examples:**
  - "`canonical/skills/aid-discover/SKILL.md` is 307 lines."
  - "The Mermaid library is 3.3 MB inlined."
  - "domain-glossary.md contains 161 terms." *(unless 161 is a contract ‚Äî usually it isn't)*
- **Stability:** every commit that touches the underlying file.
- **Inline?** **NO** ‚Äî banned from primary-category docs. Lives in `.aid/generated/metrics.md`,
  regenerated by `build-metrics.sh`.
- **Review treatment:** none. The reviewer never verifies a T3 metric; the build script
  does, and the lint confirms the build ran.

### T4 ‚Äî Temporal

Dates, cycle tags, "as of X", "verified during cycle-N", inline timestamps.

- **Examples:**
  - "Verified during cycle-25 (2026-05-25)"
  - "RESOLVED 2026-05-22"
  - "Post-merge, May 2026"
- **Stability:** every cycle. Always stale next cycle.
- **Inline?** **NO** ‚Äî banned from primary-category docs. Allowed in:
  - `STATE.md ## Review History` / `## Calibration Log` (meta-category, exempt)
  - Per-doc `changelog:` frontmatter field (exempt)
  - As a SEMANTIC ANCHOR ONLY (e.g., "post-FR2 retirement, work-003" where the date
    is a load-bearing context for the rest of the sentence) ‚Äî even then, justify
    case-by-case per [principles.md](principles.md) P1(c).
- **Review treatment:** none for exempt locations; the lint WARNS when T4 markers
  appear inline in primary-category docs.

## How tiers map to file lifecycle

| Tier | Where it lives | Lifecycle |
|------|----------------|-----------|
| T1 Concept | Inline in primary docs | Stable; edited by methodology revisions |
| T2 Structure | Inline + declared in `contracts:` frontmatter | Edited when contract changes; lint verifies against disk |
| T3 Metric | `.aid/generated/metrics.md` only | Regenerated by `build-metrics.sh` |
| T4 Temporal | Frontmatter `changelog:` + `STATE.md` history sections | Append-only; exempt from review |

## How tiers map to per-doc `kb-category:`

The kb-category determines which tiers can appear in the doc:

| kb-category | Allowed tiers | Banned tiers |
|-------------|---------------|--------------|
| **primary** + `source: hand-authored` | T1, T2 | T3, T4 (warning from lint) |
| **primary** + `source: generated` | T1, T2, T3, T4 (build script controls; lint skips content) | none |
| **meta** | T1, T2, T3, T4 (all allowed; doc is exempt anyway) | none |
| **extension** + `source: hand-authored` | T1, T2 | T3, T4 (warning from lint) |

Note: `kb-category` is per-doc role; `source` is per-doc production mode. The two are
orthogonal ‚Äî see [frontmatter-schema.md](frontmatter-schema.md). The combination
determines tier allowances and review rubric (see [review-rubric.md](review-rubric.md)).

## Recognizing tiers ‚Äî quick reference

| If a claim is... | Tier |
|------------------|------|
| "X is a kind of Y" / "X consists of A, B, C" / "X depends on Y" | T1 Concept |
| "There are exactly N X's" / "Schema requires fields A, B, C" | T2 Structure |
| "X is N lines" / "X has N functions" / "size is N bytes" | T3 Metric |
| "Verified on DATE" / "as of CYCLE-N" / "RESOLVED on DATE" | T4 Temporal |

## Edge cases

- **"There are 22 agents"** ‚Äî T2 if 22 is a contract; T3 if it's just a current
  measurement. The frontmatter `contracts:` field disambiguates: if it's listed
  there, it's T2 and the lint verifies it. Otherwise, T3 and it shouldn't be inline.
- **"The methodology document is 1,071 lines"** ‚Äî T3 (cosmetic measurement). Should
  not be inline in primary docs. Lives in metrics.md if referenced at all.
- **"work-002 (canonical-generator) shipped 2026-05"** ‚Äî T4 if it's just a "when did
  this happen" mention. T1 if "post-canonical-generator" is a load-bearing semantic
  anchor for what follows. Judgment call per P1(c).
- **"161 terms in domain-glossary"** ‚Äî T3 (cosmetic). Domain-glossary's value is the
  definitions, not the count.
- **Mermaid diagram structure** ‚Äî T1 Concept (the relationships shown). Diagram-level
  metric annotations (e.g., "5 nodes, 2.4 MB inlined") in figcaptions are T3 and
  should be cut.
- **"see `parse.sh:412`"** ‚Äî the `:412` suffix is a positional pointer that rots on the
  next edit (as volatile as a T3 metric, though it is a citation, not a measurement).
  Cite a durable anchor ‚Äî `parse.sh` plus the function/heading name ‚Äî not the line number.
  Unlike T3 counts (build-script-owned, never reviewed), a stray `:LINE` cite IS the
  reviewer's business ‚Äî governed by [principles.md](principles.md) P1(d) and flagged for
  replacement under review-rubric.md check #8.
