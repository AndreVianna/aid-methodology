# aid-config — Pipeline Configuration

Manage AID pipeline configuration. Combines what the legacy `aid-init` skill
did (first-time scaffolding) with **ongoing settings management**. Single
source of truth: `.aid/settings.yml`.

## What It Does

**First run (no `.aid/` exists)** — scaffolds the project:

1. **Classify the project** — greenfield (new) or brownfield (existing code).
2. **Collect metadata** — project name, one-line description, external doc paths, install tools, minimum grade, heartbeat interval, max parallel tasks, .gitignore policy.
3. **Scaffold the Knowledge Base** — `.aid/knowledge/` with all 16 KB document templates pre-populated for the project type.
4. **Generate meta-documents** — `STATE.md`, `README.md` (INDEX.md is auto-generated by `build-index.sh`).
5. **Write `.aid/settings.yml`** — single source of truth for all AID pipeline configuration, populated with the collected values.
6. **Set up the project context file** — `CLAUDE.md` (Claude Code) or `AGENTS.md` (Codex/Cursor) with AID placeholders. **Description is NOT duplicated here — only lives in settings.yml.**
7. **Update `.gitignore`** — based on whether the user wants `.aid/` versioned.

**Subsequent runs** — view + update settings:

- `VIEW` — render `.aid/settings.yml` grouped by section, prompt the user to select a key or exit.
- `UPDATE` — show current value, valid options, canonical default; prompt for new value; validate.
- `PERSIST` — atomic write to `.aid/settings.yml`. Re-enter VIEW so user sees the new state.
- Loop until user exits.

## When to Use

- **Once at the start** of a new project, before any other AID skill (replaces the old `aid-init` workflow).
- **Any time afterward** to view current settings or change one (minimum grade, heartbeat interval, max parallel tasks, project description, etc.).
- `--show` shows current settings and exits (read-only).
- `--reset` rewrites `.aid/settings.yml` from canonical defaults (keeps `.aid/` structure; asks for confirmation).

## Artifacts

| Artifact | Location | Purpose |
|----------|----------|---------|
| `.aid/settings.yml` | project root | **THE source of truth** for pipeline configuration; consumed by every AID skill |
| `.aid/knowledge/` | project root | All 16 KB documents + `STATE.md` + `README.md` (INDEX.md is auto-generated) |
| `{project_context_file}` | project root | `CLAUDE.md` or `AGENTS.md` with AID placeholders; project-description is NOT duplicated here |

## Settings schema

The canonical schema is at `canonical/templates/settings.yml`. Top-level sections:

| Section | Purpose |
|---|---|
| `project` | name, description, type (brownfield/greenfield) |
| `tools` | which AI host tools have AID installed |
| `review` | global default `minimum_grade` for every skill's REVIEW state |
| `execution` | `max_parallel_tasks` for `/aid-execute` + `/aid-deploy` |
| `traceability` | `heartbeat_interval` (minutes) for sub-agent visibility |

**Per-skill overrides** — any of the 9 skill names (discover, summary, interview, specify, plan, detail, execute, deploy, monitor) may be added as a top-level key with its own `minimum_grade:` to override the global value for that skill.

## How other skills read settings

Consumer skills resolve their settings using:

1. **Per-skill override** (e.g., `discover.minimum_grade`) — use if present
2. **Global category default** (`review.minimum_grade`) — otherwise
3. **Hardcoded skill default** — only if `.aid/settings.yml` is missing entirely

The canonical resolution helper is `canonical/scripts/config/read-setting.sh`.
