**Scope:**
- **Run the migration on AID's own `.aid/knowledge/`** -- the dogfood, the AC9 evidence ("AID's own
  KB migrates to the new schema"). AID is the FIRST executor of `migrate-kb-frontmatter.sh` (task-018)
  on the real corpus before any adopter runs it ([[aid-content-isolation-cornerstone]] dogfood
  discipline). Net corpus = **16 in-scope docs** (the 15 `hand-authored` primary/extension docs + the
  `host-tool-capabilities.md` `promoted from ...` edge, now in scope via task-020's widened predicate);
  `meta` + `source: generated` docs are intentionally untouched.
- Execute the load-bearing **order of operations** ([SPIKE-M3]), all committed TOGETHER so AID's CI
  never observes the strict assertion (task-020) over a mixed/un-migrated state:
  1. `migrate-kb-frontmatter.sh .aid/knowledge --propose` -> the maintainer reviews + refines the
     `.aid/.temp/kb-migration-proposal.md` worksheet at the gate: tighten each `objective:` noun-phrase,
     refine each `summary:`, and -- critically -- **confirm/correct the `sources:`** per doc (add the
     real sources the heuristic missed, drop false positives). `sources:` is NEVER auto-finalized
     (NFR-6/C4, the keystone primitive). For the pure-synthesis doc (`external-sources.md`, which
     registers NO external URLs) confirm `sources: []`.
  2. `migrate-kb-frontmatter.sh .aid/knowledge --apply` -> writes the confirmed fields into all 16
     docs, retires `intent:`, stamps `approved_at_commit:` at migration HEAD, runs the verification
     pass (each doc lint-clean). The `domain-glossary.md` body must already be on the spine shape
     (task-021) before this writes its frontmatter.
  3. **Re-run `build-kb-index.sh`** (the f002/task-015 table generator) over the migrated
     `.aid/knowledge/` and commit the regenerated `.aid/knowledge/INDEX.md`. The table now builds from
     the REAL `objective:`/`summary:`/`tags:`/`see_also:`/`audience:` (the `intent:` fallback rows
     disappear). f011 RUNS, does not edit, the generator (f002 owns it).
  4. Re-run `python .claude/skills/generate-profile/scripts/run_generator.py` and commit the
     regenerated `profiles/` (the task-018 script + the task-020 lint edit are canonical; render-drift).
  5. **Commit all of (1)-(4) together** with task-020's lint widen + AID-CI strict assertion, so AID's
     CI runs the strict check over a fully-migrated corpus -> green, never over a mixed state.
- **Verify post-migration CI-green conditions** locally before the gate: `lint-frontmatter.sh` is
  effectively hard for AID (the retained soft-skip never fires -- every AID in-scope doc carries the
  fields); the AID-CI strict assertion passes; AID's INDEX renders fully-populated rows (no `intent:`
  fallback rows); render-drift, KB-hygiene, and INDEX-fresh stay green.
- The proposal worksheet + backup tree are transient (`.aid/.temp/`, gitignored) -- NOT committed
  artifacts (C6 content-isolation; keep run-state out of the data).
- **Boundary:** this task OWNS the actual dogfood RUN of the migration on AID's real KB, the human
  `sources:` confirmation, the INDEX regen, and the commit-together sequencing. It does NOT author the
  migration script (task-018), its tests (task-019), the lint widen / ASCII guard / AID-CI assertion
  (task-020 -- this task LANDS them green by migrating first), the glossary body spine-shaping
  (task-021), or the INDEX generator (f002, task-015 -- it only re-runs it).
