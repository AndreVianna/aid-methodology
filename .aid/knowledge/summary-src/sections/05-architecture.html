How AID is built — architecture.md

Architecture

AID has a dual anatomy: a product face (building and shipping the installer) and a dogfood face (using AID to build AID). Three layers of architecture — repository, build/distribute, and process — are each governed by hard invariants.

AID's dual-face anatomy: the product face (canonical → profiles → packages) alongside the dogfood install (.claude/ + .aid/) where AID is used to build AID itself.

Layer 1

Repository Architecture

The load-bearing boundaries: canonical/ (source of truth), profiles/ (rendered output, never hand-edited), packages/ (distribution wrappers). The Emission Manifest is the safety boundary for file operations.

Layer 2

Build & Distribute Architecture

Three data flows: (1) Render: canonical → profiles via run_generator.py → VERIFY byte-compare gate. (2) Install: bootstrap (curl/irm) or package manager → AidInstallCore → user repo. (3) Pipeline: skill → agent dispatch → STATE.md writes.

Layer 3

Process Architecture

The 6-phase gated pipeline. Each skill is a state machine with states (PLAN → ACT → REVIEW → DONE). 9 agents in 3 tiers dispatch to do the work. The Knowledge Base is the gravitational center — every phase reads it; any phase can update it.

Architecture Invariants 8 rules
  1. Never edit profiles/ directly — only canonical/.
  2. The Emission Manifest is the authoritative boundary for file operations; files outside any manifest are never touched.
  3. AID content must be namespaced (aid- prefix / aid/ subtree) and never collide with user content.
  4. Shipped Bash scripts must be POSIX-compatible and ASCII-only; PowerShell must target 5.1+.
  5. The CLI and dashboard declare zero runtime dependencies.
  6. The dashboard server is loopback-only; it writes nothing and runs no LLM.
  7. Four version carriers (VERSION, package.json, pyproject.toml, git tag) must stay in sync.
  8. Every pipeline phase transition requires explicit human approval ("OK?" gate).
