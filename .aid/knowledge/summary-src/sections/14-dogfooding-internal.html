Internal — Not the Methodology

Dogfooding — Internal Repo Tooling

Not part of the AID methodology; this is how the AID repo runs AID on itself. The repo-root .claude/ is AID's own working install of itself — the same kind of tree a Claude Code adopter gets after setup.sh, but living in this repo so the maintainer can run the /aid-* skills against the AID codebase during development. It is literally what executes this very summarize pipeline. Adopters never receive this repo's .claude/; they render their own tree from canonical/. (per coding-standards.md §7a/§7d, project-structure.md, architecture.md)

Internal tooling, not a shipped spec

The dogfood .claude/ is internal to this repo and is not part of the methodology that ships to adopters. Do not confuse it with the claude-code render profile (§9), which is what adopters install. The profile is generated output; the dogfood is hand-maintained.

Hand-maintained · not generated

Outside the render set

The dogfood .claude/ is hand-maintained — NOT produced by run_generator.py. The generator writes only the 5 profile trees; .claude/ is deliberately hand-editable (the "dogfood-is-hand-editable" policy) so the maintainer can test changes in place without re-rendering. (per coding-standards.md §7a)

The 7th physical copy

Carries the bodies by hand

The byte-identity invariant the generator enforces spans 6 trees (canonical/ source + the 5 profile trees). The dogfood .claude/ carries the same byte-identical bodies by hand as a 7th physical copy, but it sits outside the enforced render set — it is not part of the 6-tree invariant. (per coding-standards.md §7a/§7d)
