Domain Glossary

AID's load-bearing vocabulary. Every term here means something specific to this project — not the generic industry sense. The Concept Spine holds the 25 native concepts the methodology is built on.

Canonical

The canonical/ directory — the single source of truth for every piece of content AID installs. Content is authored once here and rendered into five profile install trees. "Canonical" is the authority, not merely "standard".

Relates to Profile Emission Manifest Knowledge Base
Profile

One of the five rendered, host-tool-specific install trees AID targets (Claude Code, Codex CLI, Cursor, GitHub Copilot CLI, Antigravity). All five carry byte-identical skill/agent bodies; only the install root, context-file name, and agent format differ. Not a user profile or a config profile.

Relates to Canonical Emission Manifest
Work

A self-contained unit of scope created by one Interview, living at .aid/work-NNN-{slug}/. Each Work owns its own requirements, features, plan, deliveries, and tasks while sharing the project-wide Knowledge Base. The top-level pipeline scope container.

Relates to Delivery Knowledge Base Task
Delivery

An ordered, independently shippable MVP grouping of features within a Work, numbered delivery-NNN. Plan decides what goes in each delivery and in what order; Execute runs one git branch per delivery and gates each with a delivery-gate review. A strategy-level unit, not a deployment event.

Relates to Work Task Delivery Gate Execution Graph
Task

The leaf unit of execution: one agent session = one PR = one human review. Every task carries a Type from a fixed 8-value enum (RESEARCH, DESIGN, IMPLEMENT, TEST, DOCUMENT, MIGRATE, REFACTOR, CONFIGURE). The Type drives both how the executor works and how the reviewer evaluates it.

Relates to Delivery Execution Graph Task Status
Execution Graph

The dependency-and-parallel-wave plan Detail appends to PLAN.md: per-delivery tables of task precedence and the "waves" of tasks that can run in parallel. Execute consumes it to drive the parallel pool and to compute which dependents a failed task blocks. Expressed as tables, not a rendered diagram.

Aliases Wave Max Concurrent
Knowledge Base

The .aid/knowledge/ collection of structured markdown documents that holds the living understanding of a project — the gravitational center of AID. Default seed of 14 standard documents plus meta-documents (INDEX, README, STATE); configurable per project. Every phase reads it; any phase can trigger a targeted update to it.

Aliases kb-category
Concept Spine

The section of domain-glossary.md that holds a project's native load-bearing concepts, each grounded with a definition-as-used-here, a relates-to linkage, and grep-recoverable sources. It is seeded from candidate-concepts and closed by the discovery closure loop.

Relates to Knowledge Base Candidate Concepts
Emission Manifest

The declaration (canonical/EMISSION-MANIFEST.md) of exactly which files the profile renderer emits for each profile. It is the contract the VERIFY gate byte-checks the rendered profiles/ against, so canonical/ always remains the source of truth.

Aliases EmissionManifest
AidInstallCore

The shared install/update/remove engine the bootstrap installers source — Bash (lib/aid-install-core.sh) and PowerShell (lib/AidInstallCore.psm1). It holds the in-place root-agent region replacement, the diff-aware copy, and channel logic so install.sh/install.ps1 stay behaviorally identical across platforms.

Aliases Aid Install Core Protect-on-diff
AID_HOME

The mutable per-machine AID state home — ${HOME}/.aid for a normal user install (or /var/lib/aid for a shared install), overridable via the AID_HOME environment variable. Holds machine state such as registry.yml. Distinct from the install code home.

Aliases state home AID_STATE_HOME
Recipe

A pre-filled lite-path template for a recurring change pattern, under canonical/aid/recipes/, named by the change it makes (add-X / change-X / fix-X). TRIAGE matches a free-form work description to a recipe. A shortcut for known patterns, not a quality bypass — it produces the same typed, reviewed task set.

Aliases Slot
Canonical-Source Render-and-Vendor Pipeline

AID's defining build model: author once in canonical/, render to five byte-identical profiles/ via run_generator.py, prove correctness with the VERIFY byte-compare gate, then vendor into npm/PyPI wrappers and tarball per-profile bundles for GitHub Releases.

Aliases render pipeline
Dual-Face Dogfood Repository

This repository has two faces: the product (canonical/→profiles/→packages/) and the dogfood install (.claude/ rendered profile + .aid/ pipeline state/KB) where AID is used to build AID. The .aid/ content is real working state, not example data.

Relates to Canonical Knowledge Base
Polyglot Parity Obligation

Because AID must install through both Bash and PowerShell hosts and ship via both npm and PyPI, the same logic is implemented in multiple languages that MUST stay behaviorally identical — aid-install-core.sh mirrors AidInstallCore.psm1; server.mjs is a byte-parity sibling of server.py. Parity is enforced by tests, never assumed.

Relates to AidInstallCore Profile
Human-Gated Phase Advancement

The rule that the pipeline never auto-advances — a human approves every phase transition (the "OK?" gate, the Iron Man model's pilot-in-the-cockpit). Distinct from the deterministic grade gate (which is computed): this is the human checkpoint between phases.

Relates to Grade Feedback Loop
Grade

A letter score computed deterministically by grade.sh from the [SEVERITY] tags a reviewer emits — never hand-picked. The reviewer only classifies issues; the grade follows automatically (worst severity dominates, the count within it sets the modifier). A phase passes only when the grade meets the resolved minimum.

Relates to Delivery Gate Human-Gated Phase Advancement
Triage

The /aid-interview step that matches a free-form work description to a Recipe and routes the work down either the full path (every numbered phase) or the lite path (a condensed phase set, same artifacts). A confident single-target match routes to lite automatically.

Relates to Lite Path Recipe Work
Lite Path

The TRIAGE-routed condensed pipeline for small, single-target work: the same typed, reviewed artifacts as the full path but with phases collapsed (Specify+Plan+Detail fold into the interview's task breakdown). A shortcut for known patterns, not a quality bypass.

Relates to Triage Recipe
Feedback Loop

A formal pathway for a downstream phase to revise upstream artifacts, producing a traceable record (a Q&A entry in a STATE file, an IMPEDIMENT file, or a Monitor finding) rather than a silent rewrite. AID defines eleven named feedback loops; they are the design answer to drift, not a failure mode.

Relates to Human-Gated Phase Advancement Knowledge Base
Dashboard

AID's one runtime component — a read-only, loopback-only local web view that parses .aid/ pipeline state and the Knowledge Base across repos and serves them as HTML. It writes nothing and runs no LLM. The summary stage of aid-housekeep regenerates the visual KB summary it presents.

Aliases dashboard server Summary Stage
Pipeline State

The authored run-state of a Work, tracked in its STATE.md: the current phase and lifecycle, recorded phase transitions, the calibration log (agent ETA-band vs actual runtime), and — in the discovery area — the documents status table of KB-doc completeness.

Aliases Phase Transition Calibration Log
Task Status

A task's lifecycle status and its derived views: the TaskStatus enum the dashboard reader assigns, the Tasks State rollup section unioned across per-task STATE.md files, and the Task Detail per-task view. The rollup is read-time derived, never hand-written.

Aliases Tasks State TaskStatus
Delivery Gate

The per-delivery review gate recorded in delivery-NNN/STATE.md: it aggregates the deferred Quick Check Findings accumulated by Small-tier in-task quick checks and requires the delivery's grade.sh grade to meet the minimum before the delivery ships.

Aliases Delivery State Quick Check Findings
Candidate Concepts

The salience-ordered table of harvested + synthesis terms that the discovery closure loop must drive to a terminal state — each either grounded in the Concept Spine or dismissed. AID's discovery run closed 29 grounded / 80 excluded / 0 ungrounded from the initial harvest.

Aliases Ranked Candidates
