Data Model

Knowledge Base Shape

AID has no database. The Knowledge Base is a set of primary Markdown documents under .aid/knowledge/, each with a YAML frontmatter block (kb-category, source, intent, contracts, changelog). The active doc-set is declared/derived per project — not a fixed standard set — through the optional discovery.doc_set block in .aid/settings.yml; when that block is absent, a default seed is synthesized from the canonical KB templates. The INDEX.md is auto-generated from each doc's intent: field. (per schemas.md §2a, architecture.md)

Declared / derived doc-set (not a fixed standard set)

Which KB documents are in scope is declared per project via the optional discovery.doc_set block in .aid/settings.yml — a YAML block-list of pipe-delimited filename|owner|presence[:when] items. When the block is absent, synth_default_seed derives a default seed by enumerating the canonical KB templates. The doc-set size is variable per project; no fixed count is hardcoded in the dispatch logic. (per schemas.md §2a, architecture.md §Discovery run-time flow detail)

This repo's active doc-set

For the AID repo itself the confirmed set is the 15 hand-authored primary docs — architecture, project-structure, technology-stack, module-map, coding-standards, schemas, pipeline-contracts, integration-map, domain-glossary, test-landscape, tech-debt, infrastructure, feature-inventory, external-sources, plus the repo-specific custom repo-presentation — modeled cleanly as a declared carve-out rather than a fixed list (data-model → schemas, api-contracts → pipeline-contracts; security-model and ui-architecture not in scope). The generated INDEX.md indexes these alongside the two meta docs (INDEX.md, STATE.md). (per schemas.md §3, INDEX.md)

Fact stability tiers (T1–T4)

Every claim in a KB doc belongs to one of four tiers that govern whether it may appear inline. (per coding-standards.md §9a)
