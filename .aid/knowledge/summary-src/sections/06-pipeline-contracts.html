AID ships no HTTP services or RPC endpoints. Contracts here means: skill slash-command signatures and state-machine contracts, script CLI signatures and exit codes, file-format contracts, subagent dispatch conventions, and the canonical-to-5-profile renderer contract. (per pipeline-contracts.md)

File-format contracts

settings.yml

Runtime configuration

YAML 1.2, 5 mandatory top-level sections: project, tools, review, execution, traceability, plus an optional discovery section carrying the doc_set declared doc-set. Per-skill minimum_grade overrides fall back to global review.minimum_grade then the hardcoded default (A). (per schemas.md §2, §2a)

emission-manifest.jsonl

Deletion-safety boundary

JSON Lines, LF-only. Sentinel first line {"_manifest_version": 1}. Four-key record schema: profile, src, dst, sha256. The profile value is one of the 5 profile stems (claude-code, codex, cursor, copilot-cli, antigravity) — one manifest per profile. Sorted by dst for byte-stable diffs. Only files in the manifest are eligible for deletion on the next render pass. (per schemas.md §8)

STATE.md

Per-area state hub

One STATE.md per area (.aid/knowledge/STATE.md for Discovery; .aid/{work}/STATE.md per work area). Absorbs legacy per-feature and per-task STATE files (FR2 area-STATE consolidation). Sections: External Documentation, KB Documents Status, Q&A, Review History, Summarization History. (per schemas.md §3)

Heartbeat files

L3 subagent visibility

Single-line files at .aid/.heartbeat/{agent}-{unix-ts}.txt. Format: [YYYY-MM-DDTHH:MM:SSZ] STATE | progress | activity (~eta). Written with > (overwrite), never >>. Activity must change between writes. (per coding-standards.md §5a)

profiles/*.toml

Per-tool render contract (5)

TOML 1.0, parsed by aid_profile.py into the Profile dataclass. agent_format ∈ {markdown (Claude Code, Cursor, Copilot CLI, Antigravity), toml (Codex — dormant, E-CODEX-1 gated)} — the _KNOWN_AGENT_FORMATS set. The [layout], [agent]/[agent.frontmatter], [skill]/[skill.frontmatter], [filename_map], and [extras] sections — along with the RuleEntry, ExtrasConfig, LayoutConfig, FrontmatterConfig, AgentConfig, and SkillConfig dataclasses — were dropped in work-005. Surviving profile schema: top-level name, root_dir, root_file, agent_format, plus [tool_names], [model_tiers], and [capabilities]. (per schemas.md §9, module-map.md)

Declared-set → dispatch contract

The discover orchestrator resolves the active doc-set at GENERATE Step 0 and a propose→confirm checkpoint (Step 0d), then uses it to drive sub-agent dispatch data-driven from the confirmed set rather than from a hardcoded list. (per pipeline-contracts.md §Declared-Set → Dispatch Contract, architecture.md)

Mapping honors the set

Data-driven target lists

Each agent's target list = filenames where owner = <agent> in the declared set, intersected with missing-on-disk. The propose→confirm step (Step 0d) presents the inferred set as a diff against the default seed and waits for the user to confirm or edit before dispatch. (per pipeline-contracts.md, architecture.md §Step 0d)

Omission / addition / default

No-hang, dispatch-on-add, backward-compatible

No-hang on omission: a doc absent from the declared set yields an empty target list, so its owning agent is simply not dispatched. Dispatch on addition: a custom doc with an owner is appended to that agent's list. Default seed: an absent/empty discovery.doc_set falls back to synth_default_seed — prior behavior, unchanged. (per pipeline-contracts.md)
