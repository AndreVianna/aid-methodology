Source-code conventions — coding-standards.md

Coding Standards

AID's own source-code conventions — shell, PowerShell, Python, and JS/Node rules mined from the actual scripts. These apply to contributors writing AID's own implementation code (not methodology prose).

Bash / Shell

RuleDetail
Shebang#!/usr/bin/env bash
Safety flagsset -euo pipefail at top
EncodingASCII-only; LF line endings
Namingkebab-case for script filenames
File headerOne-line purpose comment below shebang

PowerShell

RuleDetail
Compatibility floorWindows PowerShell 5.1 (not pwsh 7)
EncodingASCII-only, no BOM, LF line endings
Module namingPascalCase for modules, Verb-AidNoun for functions
TLSExplicitly force TLS 1.2 (not default on 5.1)
Banned3-arg Join-Path, utf8NoBOM encoding, $IsWindows, non-ASCII characters

Python

RuleDetail
Version floor≥3.8; CI runs 3.11
Type hintsRequired on public functions
Data classesPreferred for data-carrying types
CLI argsargparse (stdlib only)
Namingsnake_case for modules and functions

Node.js / JavaScript

RuleDetail
Module formatESM (.mjs extension)
Node importsUse node: scheme (import fs from 'node:fs')
Version floor≥Node 18; CI runs 20
No runtime depsstdlib only for dashboard server

Exit Codes

CodeMeaning
0Success
1General / runtime error
2Invocation error (bad arguments, missing required input)
3Precondition not met (e.g. not an AID repo)
4Conflict / collision (e.g. protect-on-diff triggered)
6Format version mismatch (migration required)

Security conventions

All downloaded payloads (curl/irm installs) must verify a SHA-256 checksum before executing. No hardcoded credentials. All user-controlled paths must be validated (no traversal). Network requests use TLS only.
