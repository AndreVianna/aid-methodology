What AID does — capability-inventory.md

Capability Inventory

AID is a methodology delivered as a multi-profile CLI installer. Its capabilities fall into four groups: the pipeline skills you drive, the on-demand skills you invoke when needed, the CLI that installs and maintains it, and the multi-tool distribution that ships it to five host agents.

Pipeline Skills — the lifecycle

9 skills that map to the 12-phase pipeline. Each skill is a state machine: the user invokes it, the agent works through states (PLAN → ACT → REVIEW → DONE), and the human approves advancement at every gate.

Phase: Init / Config Configure the pipeline /aid-config
What
View or set project settings; scaffolds .aid/settings.yml and the KB state file.
When
Before starting work on a new repository, or to adjust pipeline settings.
Phase: Discover Discover an existing project /aid-discover
What
Analyzes a repository and builds the Knowledge Base, with a built-in review → Q&A → fix → approval gate.
When
Brownfield projects run this first. Generalized to any-domain work-guide generator (v1.1.1+).
Phase: Interview Gather requirements /aid-interview
What
Adaptive one-question-at-a-time interview producing REQUIREMENTS.md, then decomposes them into feature files.
When
After discovery, when starting a new Work. Includes TRIAGE for lite-path routing.
Phase: Specify Specify a feature /aid-specify
What
Collaboratively writes a technical SPEC.md, one feature at a time.
When
Full path only; skipped on lite path. After interview.
Phase: Plan Plan deliveries /aid-plan
What
Sequences feature specs into deliverables, each a functional MVP building on the last.
When
Full path only. After specification.
Phase: Detail Break down into tasks /aid-detail
What
Decomposes deliverables into small, typed, dependency-ordered tasks with an execution graph.
When
Full path only. After planning.
Phase: Implement / Review / Test Execute a task /aid-execute
What
Runs a task by type (Implement/Test/Refactor/etc.) with a built-in review → fix loop; one git branch per delivery.
When
For every task in a delivery, after detailing.
Phase: Deploy Ship a release /aid-deploy
What
Packages completed deliveries, verifies the combined build, generates release notes.
When
When deliveries are ready to ship.
Phase: Track / Triage Observe production /aid-monitor
What
Interprets telemetry, classifies findings, and routes bugs / change requests back into the pipeline.
When
After deployment, to close the feedback loop from production.

On-Demand Skills

4 skills run outside the linear pipeline, when the user needs them.

On-demand Ask the KB a question /aid-query-kb
What
Answers a free-form question grounded in the KB, the live code, and in-flight work; cites sources or names the gap.
When
Any time a grounded, KB-sourced answer is needed.
On-demand Targeted KB update /aid-update-kb
What
Applies a described change to the KB through the same review/approval gate as discovery.
When
When a specific KB document needs correcting or extending between full discovery runs.
On-demand Housekeeping /aid-housekeep
What
Re-discovers changed KB docs, regenerates the visual summary, and sweeps stale work artifacts.
When
Periodically, or after significant code changes to resync the KB.
On-demand Visual KB summary /aid-summarize
What
Generates a single-file kb.html — a visually rich, newcomer-friendly view of the Knowledge Base. This page was generated by this skill.
When
To share KB content with non-technical stakeholders or review KB completeness visually.

CLI Installer

AID installs into a repository as content namespaced under an aid- prefix and isolated from user content. The CLI (entry point under bin/) provides these operations:

CLI

Install

Adds AID content to a repository (and a per-machine AID state home at ~/.aid). Available via npm, PyPI, curl, or irm.

CLI

Update

Brings an installed repository up to the current AID version, migrating the format when needed.

CLI

Remove

Cleans AID content back out, bounded by the manifest — only removes files it installed.

CLI

Dashboard

Serves a local web dashboard that reads work-tracking state across repositories. Loopback-bound. No LLM calls.

Multi-Tool Distribution

The same capabilities are authored once in canonical/ and rendered into five host-agent install profiles so a user on any of these tools gets the identical methodology.

The canonical-source render-and-vendor pipeline: author once in canonical/, render to 5 profiles, vendor into npm + PyPI + GitHub Releases. The VERIFY byte-compare gate ensures profiles always match canonical.

What AID does NOT do

AID is not a runtime framework or library you import into an application — it is a process methodology plus the tooling that installs and runs it. It does not host services or store project data remotely; the dashboard runs locally on loopback only.
