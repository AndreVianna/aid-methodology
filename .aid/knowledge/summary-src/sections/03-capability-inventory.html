What AID does — capability-inventory.md

Capability Inventory

AID is a methodology delivered as a multi-profile CLI installer. Its capabilities fall into four groups: the pipeline skills you drive, the on-demand skills you invoke when needed, the CLI that installs and maintains it, and the multi-tool distribution that ships it to five host agents.

Pipeline Skills — the lifecycle

10 skills that drive the 6-phase pipeline plus its one-time bootstrap and optional delivery steps. Each skill is a state machine: the user invokes it, the agent works through states (PLAN → ACT → REVIEW → DONE), and the human approves advancement at every gate. Phase 2 (Interview) is realized by two chained skills: /aid-describe (Phase 2a) and /aid-define (Phase 2b).

Setup: Config (bootstrap) Configure the pipeline /aid-config
What
View or set project settings; scaffolds .aid/settings.yml and the KB state file.
When
Before starting work on a new repository, or to adjust pipeline settings.
Phase 1: Discover Discover an existing project /aid-discover
What
Analyzes a repository and builds the Knowledge Base, with a built-in review → Q&A → fix → approval gate.
When
Brownfield projects run this first. Generalized to any-domain work-guide generator (v1.1.1+).
Phase 2a: Describe Gather requirements /aid-describe
What
Adaptive one-question-at-a-time interview driven by the seasoned-analyst elicitation engine (NFR-7 envelope + five-step selector), producing an approved REQUIREMENTS.md. Runs TRIAGE (full vs. lite routing) and, on greenfield projects, forward-authors a KB seed.
When
After discovery (or directly for greenfield), when starting a new Work. The aid-interviewer agent conducts the dialogue.
Phase 2b: Define Decompose into features /aid-define
What
Decomposes the approved REQUIREMENTS.md into per-feature SPEC.md stubs, then cross-references the requirements and feature boundaries against the KB and codebase.
When
Full path only; immediately after /aid-describe completes. Hands off to /aid-specify.
Phase 3: Specify Specify a feature /aid-specify
What
Collaboratively writes a technical SPEC.md, one feature at a time.
When
Full path only; skipped on lite path. After interview.
Phase 4: Plan Plan deliveries /aid-plan
What
Sequences feature specs into deliverables, each a functional MVP building on the last.
When
Full path only. After specification.
Phase 5: Detail Break down into tasks /aid-detail
What
Decomposes deliverables into small, typed, dependency-ordered tasks with an execution graph.
When
Full path only. After planning.
Phase 6: Execute Execute a task /aid-execute
What
Runs a task by type (Implement/Test/Refactor/etc.) with a built-in review → fix loop; one git branch per delivery.
When
For every task in a delivery, after detailing.
Deliver: Deploy (optional) Ship a release /aid-deploy
What
Packages completed deliveries, verifies the combined build, generates release notes.
When
When deliveries are ready to ship.
Deliver: Monitor (optional) Observe production /aid-monitor
What
Interprets telemetry, classifies findings, and routes bugs / change requests back into the pipeline.
When
After deployment, to close the feedback loop from production.

Requirements-Gathering Capabilities (deep dive)

Three capabilities distinguish /aid-describe from a simple question list: the seasoned-analyst elicitation engine, greenfield KB seed authoring, and the downstream conformance check.

NFR-7 Elicitation Engine

Seasoned-analyst questions

Every question carries a concrete Suggested: answer and a grounded Why: rationale. The five-step next-move selector picks the most useful next question each turn instead of following a fixed script. Calibration adapts depth and vocabulary to the user's expertise level.

Greenfield Inversion

Forward-authored KB seed

For new projects with no existing code, /aid-describe authors a 5-element KB seed from elicited intent (concept-spine, architecture, conventions, tech-stack, decisions), each stamped source: forward-authored. The design docs become the source of truth; code is built to conform to them.

Build Conformance

Flag divergence, never overwrite

/aid-housekeep's Conformance Lane compares as-built code against each forward-authored design doc in the code→design direction. Any divergence is flagged for human reconciliation — the design doc is never auto-overwritten with as-built reality.

On-Demand Skills

4 skills run outside the linear pipeline, when the user needs them.

On-demand Ask the KB a question /aid-query-kb
What
Answers a free-form question grounded in the KB, the live code, and in-flight work; cites sources or names the gap.
When
Any time a grounded, KB-sourced answer is needed.
On-demand Targeted KB update /aid-update-kb
What
Applies a described change to the KB through the same review/approval gate as discovery.
When
When a specific KB document needs correcting or extending between full discovery runs.
On-demand Housekeeping /aid-housekeep
What
Re-discovers changed KB docs, regenerates the visual summary, and sweeps stale work artifacts.
When
Periodically, or after significant code changes to resync the KB.
On-demand Visual KB summary /aid-summarize
What
Generates a single-file kb.html — a visually rich, newcomer-friendly view of the Knowledge Base. This page was generated by this skill.
When
To share KB content with non-technical stakeholders or review KB completeness visually.

CLI Installer

AID installs into a repository as content namespaced under an aid- prefix and isolated from user content. The CLI (entry point under bin/) provides these operations:

CLI

Install

Adds AID content to a repository (and a per-machine AID state home at ~/.aid). Available via npm, PyPI, curl, or irm.

CLI

Update

Brings an installed repository up to the current AID version, migrating the format when needed.

CLI

Remove

Cleans AID content back out, bounded by the manifest — only removes files it installed.

CLI

Dashboard

Serves a local web dashboard that reads work-tracking state across repositories. Loopback-bound. No LLM calls.

Multi-Tool Distribution

The same capabilities are authored once in canonical/ and rendered into five byte-identical host-agent install profiles. All five are bundled into one release payload that is published identically to every distribution channel — a user installs from any one channel and the installer drops the profile matching their tool. The channels are equivalent copies, not per-profile outputs.

The canonical-source render-and-distribute pipeline: author once in canonical/, render to 5 byte-identical profiles (guarded by the VERIFY byte-gate), bundle all five into one release payload, then publish that same payload to npm, PyPI, and GitHub Releases. A user installs from any one channel and the installer drops the profile matching their tool — the channels are equivalent copies, not per-profile outputs.

What AID does NOT do

AID is not a runtime framework or library you import into an application — it is a process methodology plus the tooling that installs and runs it. It does not host services or store project data remotely; the dashboard runs locally on loopback only.
