The Pipeline

AID defines six sequential development phases — from brownfield discovery through reviewed execution — with explicit human-controlled phase gates and feedback loops. Two optional Deliver skills (Deploy, Monitor) sit at the end of the pipeline, run on demand. Each numbered phase maps to a slash-command skill. (per architecture.md, repo-presentation.md)

The development pipeline (Discover → Execute) is sequential by default: human approval is required to advance between phases. The optional skills — Deploy and Monitor (end-of-pipeline Deliver), plus the read-only Summarize, the off-pipeline Housekeep, the read-only project Q&A helper Query-KB, and the targeted KB update skill Update-KB — run on demand; none is required and Deploy/Monitor do not presuppose each other. Feedback loops exist as escape hatches — Monitor routes both bugs and change requests back to Interview, which triages them; Execute can issue IMPEDIMENT escalations; any phase can revisit Discovery, and Housekeep reconciles the KB back into Discovery. After Interview, a Lite path decision routes small work directly to task execution, bypassing Specify, Plan, and Detail. (per feature-inventory.md §Engineering features)

Phase gate philosophy

Each skill exits after one state and re-enters on the next slash-command invocation. No auto-advance — the human controls every transition. This is intentional: the human is the Director, not the scheduler. (per architecture.md §Thin-Router state machine)
