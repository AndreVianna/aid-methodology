Tech Debt

Severity-classified open technical and methodology debt in AID. Read this before starting work in any area — declared debt items and non-obvious gotchas may change your approach or scope.

H1 — Five install manifests must stay lockstep
Severity: HIGH · Area: Distribution

The five install manifests (npm package.json, PyPI pyproject.toml, install.sh, install.ps1, AidInstallCore) must all be updated together when any installed file set changes. A single missed update silently ships a broken install for one channel. Currently managed by manual discipline; a lockstep guard is needed.

M1 — npm and PyPI channels effectively blocked
Severity: MEDIUM · Area: Distribution

Versions 1.1.0 and 1.1.1 are published on npm and PyPI but the channels have outstanding issues (1.1.0 should be deprecated; stale release notes). Affects discoverability and trust.

M2 — Heavy CI gates run on master only
Severity: MEDIUM · Area: Testing

The canonical test suite (tests/run-all.sh) and docs site build run only on push/PR to master. Feature branches skip them, so a direct-to-master merge can break master in multiple ways. Run locally before merging.

M3 — Stale docs/repository-structure.md
Severity: MEDIUM · Area: Documentation

The docs site's repository-structure.md drifted from the actual repo layout. Out of discovery scope to fix; tracked here to prevent it being used as a reference.

L1 — Dead code in install.sh
Severity: LOW · Area: Distribution

The OVERALL_BLOCKED/exit 5/.aid-new protect-on-diff branch in install.sh is unreachable (aid-install-core eliminated the .aid-new path). Safe to remove.

L2 — --sign flag deferred
Severity: LOW · Area: Security

Code-signing for release bundles is planned but not yet implemented. The release pipeline produces unsigned bundles.

L3 — Legacy install path in tests
Severity: LOW · Area: Testing

Some older test fixtures still reference the legacy pre-1.0.0 install layout. They pass but test the wrong path for new-format repos.

L4 — No coverage % metric
Severity: LOW · Area: Testing

Test coverage is assessed by presence/absence of suites, not line percentage. A coverage metric tool would help identify untested paths.
