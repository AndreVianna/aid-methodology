Infrastructure

How AID moves from source to running/published software: source control, CI/CD, the multi-profile render, the three distribution channels, versioning, the bootstrap installers, and the dashboard runtime.

Release Pipeline

1
Quality gateAll CI checks pass; run-all.sh green locally; render-drift clean
2
Version bumpBump all 4 version carriers: VERSION, package.json, pyproject.toml, git tag
3
GitHub releaserelease.yml workflow creates release with 9 assets (per-profile tarballs + install scripts + checksums)
4
npm publishParallel to PyPI; publishes aid-installer@1.1.1
5
PyPI publishParallel to npm; publishes aid-installer 1.1.1

Distribution Channels

ChannelBootstrap commandNotes
npm npx aid-installer Node ≥18 required; zero runtime deps
PyPI pip install aid-installer && python -m aid_installer Python ≥3.8 required; zero runtime deps
Bash bootstrap curl -fsSL .../install.sh | bash No runtime deps; SHA-256 verified
PowerShell bootstrap irm .../install.ps1 | iex PS 5.1+; TLS 1.2 explicit; SHA-256 verified
Offline tarball Per-profile tarballs from GitHub Releases 9 assets per release

Versioning

Four version carriers (must stay in sync)

VERSION (plain text) · packages/npm/package.json · packages/pypi/pyproject.toml · git tag. Current: v1.1.1. The format_version in .aid/settings.yml is independent — it tracks the install schema, not the AID CLI version.

Dashboard

localhost only

The dashboard server (server.mjs or server.py) binds to 127.0.0.1 only. Multi-repo view of pipeline state + KB summary. Exposed privately via Tailscale if needed; never on the public internet.
