Overview

What is AID?

AID installs a structured AI workflow into any repository. It tells AI agents how to discover a project, gather requirements, write specs, plan work, execute tasks, and ship releases — one gated phase at a time, with a human in the loop at every step.

The Problem

AI without structure ships hallucination

Dropping an AI agent into a codebase without a Knowledge Base produces technically plausible but architecturally wrong code. AID solves this with Discovery-first: build the KB before writing a line of spec.

The Solution

Waterfall discipline at AI speed

AID revives Waterfall's sequential rigour — understand, specify, plan, build, verify, ship — but executes each phase with AI agents so the economics flip: discovery takes hours, not months.

The Delivery

A CLI installer for 5 host tools

AID ships as a multi-profile CLI installer. Install into a repo once; get 13 skills, 9 agents, and the full pipeline ready inside Claude Code, Codex, Cursor, GitHub Copilot CLI, or Antigravity.

AID 6-phase pipeline: Discover → Interview → Specify → Plan → Detail → Execute. Every phase transition requires human approval. (aid-config bootstraps before the pipeline; Deploy and Monitor are optional Deliver skills after it.)

Domain Glossary

AID's load-bearing vocabulary. Every term here means something specific to this project — not the generic industry sense. The Concept Spine holds the 25 native concepts the methodology is built on.

Canonical

The canonical/ directory — the single source of truth for every piece of content AID installs. Content is authored once here and rendered into five profile install trees. "Canonical" is the authority, not merely "standard".

Relates to Profile Emission Manifest Knowledge Base
Profile

One of the five rendered, host-tool-specific install trees AID targets (Claude Code, Codex CLI, Cursor, GitHub Copilot CLI, Antigravity). All five carry byte-identical skill/agent bodies; only the install root, context-file name, and agent format differ. Not a user profile or a config profile.

Relates to Canonical Emission Manifest
Work

A self-contained unit of scope created by one Interview, living at .aid/work-NNN-{slug}/. Each Work owns its own requirements, features, plan, deliveries, and tasks while sharing the project-wide Knowledge Base. The top-level pipeline scope container.

Relates to Delivery Knowledge Base Task
Delivery

An ordered, independently shippable MVP grouping of features within a Work, numbered delivery-NNN. Plan decides what goes in each delivery and in what order; Execute runs one git branch per delivery and gates each with a delivery-gate review. A strategy-level unit, not a deployment event.

Task

The leaf unit of execution: one agent session = one PR = one human review. Every task carries a Type from a fixed 8-value enum (RESEARCH, DESIGN, IMPLEMENT, TEST, DOCUMENT, MIGRATE, REFACTOR, CONFIGURE). The Type drives both how the executor works and how the reviewer evaluates it.

Execution Graph

The dependency-and-parallel-wave plan Detail appends to PLAN.md: per-delivery tables of task precedence and the "waves" of tasks that can run in parallel. Execute consumes it to drive the parallel pool and to compute which dependents a failed task blocks. Expressed as tables, not a rendered diagram.

The .aid/knowledge/ collection of structured markdown documents that holds the living understanding of a project — the gravitational center of AID. Because AID must install through both Bash and PowerShell hosts and ship via both npm and PyPI, the same logic is implemented in multiple languages that MUST stay behaviorally identical — aid-install-core.sh mirrors AidInstallCore.psm1; server.mjs is a byte-parity sibling of server.py. Parity is enforced by tests, never assumed.

Human-Gated Phase Advancement

The rule that the pipeline never auto-advances — a human approves every phase transition (the "OK?" gate, the Iron Man model's pilot-in-the-cockpit). Distinct from the deterministic grade gate (which is computed): this is the human checkpoint between phases.

Relates to Grade Feedback Loop
Grade

A letter score computed deterministically by grade.sh from the [SEVERITY] tags a reviewer emits — never hand-picked. The reviewer only classifies issues; the grade follows automatically (worst severity dominates, the count within it sets the modifier). A phase passes only when the grade meets the resolved minimum.

Relates to Delivery Gate Human-Gated Phase Advancement
Triage

The /aid-interview step that matches a free-form work description to a Recipe and routes the work down either the full path (every numbered phase) or the lite path (a condensed phase set, same artifacts). A confident single-target match routes to lite automatically.

Relates to Lite Path Recipe Work
Lite Path

The TRIAGE-routed condensed pipeline for small, single-target work: the same typed, reviewed artifacts as the full path but with phases collapsed (Specify+Plan+Detail fold into the interview's task breakdown). A shortcut for known patterns, not a quality bypass.

Relates to Triage Recipe
Feedback Loop

A formal pathway for a downstream phase to revise upstream artifacts, producing a traceable record (a Q&A entry in a STATE file, an IMPEDIMENT file, or a Monitor finding) rather than a silent rewrite. AID defines eleven named feedback loops; they are the design answer to drift, not a failure mode.

Relates to Human-Gated Phase Advancement Knowledge Base
Dashboard

AID's one runtime component — a read-only, loopback-only local web view that parses .aid/ pipeline state and the Knowledge Base across repos and serves them as HTML. It writes nothing and runs no LLM. The summary stage of aid-housekeep regenerates the visual KB summary it presents.

Design Rationale — decisions.md

Decisions

The 18 load-bearing design, process, and tooling decisions that shaped AID. Each records what was decided, why, and what was rejected — so a newcomer can reconstruct the rationale the artifacts alone cannot show.

D1 Waterfall sequence with AI execution Accepted
Decision Adopt the Waterfall sequence (Understand → Specify → Plan → Build → Verify → Ship) deliberately, executed by AI at AI speed.
Why Waterfall failed because humans could not execute it fast enough to afford iteration; AI changes the economics (discovery in hours, specs in minutes, iteration costs tokens not sprints).
Rejected
Pure Agile / skip-to-implementation — omits the structure that Agile iterations skip because it felt too slow.
D2 Knowledge before specification (Discovery-first) Accepted
Decision A dedicated Discovery phase produces a Knowledge Base before any spec is written; brownfield projects must run it.
Why Dropping an agent into a brownfield codebase without a KB produces hallucination — "technically plausible but architecturally wrong code".
Rejected
Spec-first (SDD) — "incomplete": it assumes you already understand the system.
D3 Spec as hypothesis + eleven formal feedback loops Accepted
Decision Treat specs as living, revisable artifacts; define 11 named feedback loops so any phase can revise an upstream artifact with a traceable revision history.
Why Any spec written before implementation is partially wrong; the choice is whether to revise formally (audit trail) or informally (silent workarounds / hidden debt).
Rejected
Re-spec-from-scratch on change — lossy and untraceable.
D4 RAG by convention (no vector database) Accepted
Decision Retrieve context via a 3-tier convention (always-loaded INDEX.md → one KB doc on demand → exact path:line citation) instead of embeddings.
Why The KB is small (18 short markdown docs); the bottleneck is knowing what exists, not retrieval speed. Convention beats infrastructure.
Rejected
Vector DB / embeddings / chunking — "at a fraction of the operational complexity".
D5 Deterministic computed grade (not LLM-judged) Accepted
Decision The reviewer classifies issues by bracketed severity ([CRITICAL]..[MINOR]); the letter grade is computed by grade.sh (worst severity dominates; count sets modifier). The reviewer never hand-picks a grade.
Why Determinism makes progress visible (D→C means all highs fixed), enables loop detection (same grade 3 cycles = systemic), and applies one rubric everywhere.
Rejected
Reviewer-assigned letter grade — non-reproducible and gameable.
D6 Adversarial review separation (reviewer tier ≥ executor) Accepted
Decision The agent that writes never grades its own work; the reviewer's model tier is always ≥ the executor's and runs in a clean context after the executor finishes.
Why Prevents the reviewer anchoring on the executor's framing; catches spec, architecture, and convention issues tests cannot detect.
Rejected
Single-agent self-review and "tests pass = done" — an explicit anti-pattern.
D7 Single canonical source rendered to five profiles + VERIFY gate Accepted
Decision Author once in canonical/, render to five tool-specific profiles/ trees with byte-identical bodies, guarded by a deterministic byte-compare VERIFY gate and a CI render-drift check.
Why Five host tools must stay in lockstep; one source eliminates five-way divergence.
Rejected
Hand-maintaining per-tool copies — eliminated by "never edit profiles/ directly" rule.
D8 Emission-manifest format = JSON Lines Accepted
Decision Record every emitted file (profile/src/dst/sha256) one-per-line in a .jsonl manifest, sorted by dst, LF-only, with a {"_manifest_version": 1} sentinel.
Why Streamable, greppable, diff-friendly, deterministic, and zero runtime dependency (Python stdlib json).
Rejected
TOML — verbose, no streaming, hard to diff multi-line records.
YAML — ordering/quoting ambiguity hurts determinism.
SQLite — binary, not diff-friendly, adds a runtime dependency.
D9 Pure-mirror deletion bounded by the manifest Accepted
Decision On re-render, the generator deletes only paths in the previous manifest but absent from the current; files outside any manifest are never touched.
Why Self-cleaning installs/updates must prune renamed/dropped AID files without ever risking user-authored files — the manifest is the authoritative safety boundary.
Rejected
Unbounded directory prune / "delete whatever isn't re-emitted" — could delete user content.
D10 Polyglot, dual-channel, zero-dependency distribution Accepted
Decision Implement install logic in both Bash and PowerShell; publish through both npm and PyPI (plus curl/irm bootstrap and offline tarballs); declare zero runtime dependencies.
Why AID must bootstrap on bare Bash and bare PowerShell hosts and reach both Node and Python users; zero deps keeps install fast and supply-chain-light.
Rejected
Single-language / single-channel install — would exclude whole host populations.
D11 Content isolation + in-place root-file markers Accepted
Decision Namespace all AID content (skills/agents carry aid- prefix); update root context files (CLAUDE.md/AGENTS.md) in place only between <!-- AID:BEGIN --> / <!-- AID:END --> markers.
Why AID content and user content must never collide, and aid update must be able to prune AID's own stale files in place with no risk to user files.
Rejected
The old .aid-new sidecar file — superseded and removed in v1.1.0.
D12 CLI install scope: cwd-driven, no scan, CODE/STATE split Settled, partial impl
Decision aid <cmd> acts on the current directory's repo (like git); no machine-wide filesystem scan. Internal homes split into read-only AID_CODE_HOME (self-located payload) and mutable AID_STATE_HOME (${AID_HOME}).
Why A real-machine dogfood exposed that writing machine state into a root-owned $AID_HOME failed for unprivileged aid, and a $HOME-only scan missed repos under /srv/projects.
Rejected
An install-time scope marker — self-correcting runtime probe is better.
A $HOME-walking scan — misses out-of-home repos, scans wrong $HOME under sudo.
D13 Per-repo format_version stamp (git model) Settled
Decision Each repo records format_version: <int> in its .aid/settings.yml, decoupled from CLI semver; a newer stamp = refuse to operate (fail-safe), older/absent = offer migration.
Why A repo's migration state is a property of the repo — it must live with the repo (git's core.repositoryformatversion model).
Rejected
A machine-level $AID_HOME/.migrated marker — can't be written by unprivileged aid and doesn't travel with the repo.
D14 Lite path + description-first TRIAGE Accepted
Decision Every interview opens with TRIAGE: the user describes the work in plain language, the agent infers the work-type and best-matching recipe, and a confident match routes to a condensed lite path that skips Specify/Plan/Detail.
Why Proportionality should be automated, not weighed per change; most individual tasks are small and should not pay full-pipeline overhead.
Rejected
One-size-fits-all full pipeline — over-heavy for small work, an explicit anti-pattern.
A user-picked work-type menu — "You never pick this from a menu".
D15 Nine agents in three tiers (role consolidation) Accepted
Decision 9 specialist agents across Large/Medium/Small tiers, mapped per profile to concrete models. Large (4): architect, developer, reviewer, tech-writer. Medium (4): researcher, designer, tester, deployer. Small (1): clerk.
Why Match model cost to task stakes; one mechanical Small-tier aid-clerk avoids Large-tier overhead for glob/extract work.
Rejected
Former larger roster with five separate discovery-* agents — collapsed into aid-researcher instances.
D16 PowerShell 5.1 compatibility floor Accepted
Decision All shipped PowerShell must run on Windows PowerShell 5.1 (a fresh Windows box has 5.1, not pwsh 7), enforced by an AST lint and a real 5.1 CI lane.
Why The bare-box bootstrap promise ("5.1+") means the installer must work before the user installs anything newer.
Rejected
Requiring pwsh 7 / using 7-only constructs (TLS1.2 defaults, 3-arg Join-Path, utf8NoBOM, $IsWindows).
D17 Prose over scripts in skill definitions Accepted
Decision Trivial state/argument handling is done in SKILL.md prose, not bash; only non-trivial, reused, deterministic operations are extracted to canonical/aid/scripts/.
Why Skills are executed by an AI host that reads prose; inventing script infrastructure for trivial work adds maintenance cost without value.
Rejected
Scripting every step — over-engineering.
D18 KB forbids diagrams; the HTML summary embraces them Accepted
Decision KB documents use A -> B arrows, relationship tables, and numbered flow lists — no diagrams. The aid-summarize HTML viewer (kb.html) is the opposite: a visually rich, newcomer-facing product where more concept infographics are better.
Why KB docs are dual-audience (human + machine) and must stay grep-able and diff-stable; the summary is a non-technical-newcomer presentation artifact with a different goal.
Rejected
Diagrams in KB docs — hurt machine-readability/diffability.
Prior Mermaid-engine approach in the summary — superseded by pre-rendered inline SVG (dropping ~3MB engine) plus a Playwright visual-fidelity gate.
What AID does — capability-inventory.md

Capability Inventory

AID is a methodology delivered as a multi-profile CLI installer. Its capabilities fall into four groups: the pipeline skills you drive, the on-demand skills you invoke when needed, the CLI that installs and maintains it, and the multi-tool distribution that ships it to five host agents.

Pipeline Skills — the lifecycle

9 skills that drive the 6-phase pipeline plus its one-time bootstrap and optional delivery steps. Each skill is a state machine: the user invokes it, the agent works through states (PLAN → ACT → REVIEW → DONE), and the human approves advancement at every gate.

CLI Installer

AID installs into a repository as content namespaced under an aid- prefix and isolated from user content. The CLI (entry point under bin/) provides these operations:

CLI

Install

Adds AID content to a repository (and a per-machine AID state home at ~/.aid). Available via npm, PyPI, curl, or irm.

CLI

Update

Brings an installed repository up to the current AID version, migrating the format when needed.

CLI

Remove

Cleans AID content back out, bounded by the manifest — only removes files it installed.

CLI

Dashboard

Serves a local web dashboard that reads work-tracking state across repositories. Loopback-bound. No LLM calls.

Multi-Tool Distribution

The same capabilities are authored once in canonical/ and rendered into five byte-identical host-agent install profiles. All five are bundled into one release payload that is published identically to every distribution channel — a user installs from any one channel and the installer drops the profile matching their tool. The channels are equivalent copies, not per-profile outputs.

The canonical-source render-and-distribute pipeline: author once in canonical/, render to 5 byte-identical profiles (guarded by the VERIFY byte-gate), bundle all five into one release payload, then publish that same payload to npm, PyPI, and GitHub Releases. A user installs from any one channel and the installer drops the profile matching their tool — the channels are equivalent copies, not per-profile outputs.

What AID does NOT do

AID is not a runtime framework or library you import into an application — it is a process methodology plus the tooling that installs and runs it. It does not host services or store project data remotely; the dashboard runs locally on loopback only.

How AID is built — architecture.md

Architecture

AID has a dual anatomy: a product face (building and shipping the installer) and a dogfood face (using AID to build AID). Three layers of architecture — repository, build/distribute, and process — are each governed by hard invariants.

AID's dual-face anatomy: the product face (canonical → profiles → packages) alongside the dogfood install (.claude/ + .aid/) where AID is used to build AID itself.

Layer 1

Repository Architecture

The load-bearing boundaries: canonical/ (source of truth), profiles/ (rendered output, never hand-edited), packages/ (distribution wrappers). The Emission Manifest is the safety boundary for file operations.

Layer 2

Build & Distribute Architecture

Three data flows: (1) Render: canonical → profiles via run_generator.py → VERIFY byte-compare gate. (2) Install: bootstrap (curl/irm) or package manager → AidInstallCore → user repo. (3) Pipeline: skill → agent dispatch → STATE.md writes.

Layer 3

Process Architecture

The 6-phase gated pipeline. Each skill is a state machine with states (PLAN → ACT → REVIEW → DONE). 9 agents in 3 tiers dispatch to do the work. The Knowledge Base is the gravitational center — every phase reads it; any phase can update it.

Architecture Invariants 8 rules
  1. Never edit profiles/ directly — only canonical/.
  2. The Emission Manifest is the authoritative boundary for file operations; files outside any manifest are never touched.
  3. AID content must be namespaced (aid- prefix / aid/ subtree) and never collide with user content.
  4. Shipped Bash scripts must be POSIX-compatible and ASCII-only; PowerShell must target 5.1+.
  5. The CLI and dashboard declare zero runtime dependencies.
  6. The dashboard server is loopback-only; it writes nothing and runs no LLM.
  7. Four version carriers (VERSION, package.json, pyproject.toml, git tag) must stay in sync.
  8. Every pipeline phase transition requires explicit human approval ("OK?" gate).

How to write AID artifacts — authoring-conventions.md

Authoring Conventions

AID mandates specific conventions for authoring its own artifacts: KB documents, reviewer ledgers, skill prose, and content-isolated host files. Read this before writing any AID methodology artifact.

KB Document Layout

Every KB doc follows this structure, in order:

1
YAML frontmatterkb-category, source, objective, summary, sources, tags, see_also, audience
2
Title (H1)Matches the filename slug
3
Index / ContentsH2 anchors to all major sections
4
Content sectionsDual-audience (human + machine); no diagrams
5
Change LogAppend-only revision history table

Dual-Audience Standard

KB docs serve two readers simultaneously:

  • Humans: single-concern (one doc, one topic), junior-clear language, tables over prose lists, no diagrams (use A -> B arrows and numbered flows instead)
  • Machines: classified (frontmatter tags for RAG routing), durable anchors (id attributes for citation), frontmatter → index → content → changelog structure

No diagrams rule in KB docs

KB documents use relationship tables and numbered flow lists — no Mermaid, no SVG. The HTML summary (this page) is the exception: it is the newcomer-facing visual layer where infographics are deliberately embraced (D18).

Reviewer Ledger Schema

All review output uses a 7-column markdown table (the "reviewer ledger"):

#SeverityStatusDocLineDescriptionEvidence
1[CRITICAL]Pendingfile.md42Example finding descriptionQuote or reference

Severity tags must be bracketed (e.g. [CRITICAL], [HIGH], [MEDIUM], [LOW], [MINOR]) for grade.sh to count them. Sentence-case tags are counted as zero. Status enum: Pending / Fixed / Recurred / Accepted / OOS / Invalid. No narrative sections in ledgers.

Content Isolation Rules

Naming

aid- prefix

All AID skills, agents, scripts, and templates carry the aid- prefix or live under an aid/ subtree. Never use bare names that could collide with user content.

Root files

AID:BEGIN / AID:END

Root agent-context files (CLAUDE.md, AGENTS.md) are updated in place only between <!-- AID:BEGIN --> and <!-- AID:END --> markers. Never the full file.

Cleanup

Manifest-bounded prune

Orphan cleanup deletes only files that were in the previous Emission Manifest but absent from the current. Files outside any manifest are never touched.

The Eleven Feedback Loops

Any phase can revise upstream artifacts through a named feedback loop, producing a traceable record rather than a silent rewrite. AID defines 11 named loops (abbreviated names from pipeline-contracts.md):

L1

Discover → KB Update

Discovery findings require updating an existing KB document.

L2

Interview → Discover

Q&A reveals a KB gap; targeted update triggered before spec.

L3

Specify → Requirements

Specification reveals an ambiguity or missing requirement.

L4

Plan → Specify

Planning reveals an under-specified feature; spec revised.

L5

Detail → Plan

Task breakdown reveals delivery sequencing needs adjustment.

L6

Execute → Detail

Implementation reveals a task that needs re-scoping or splitting.

L7

Execute → Specify

Spec assumption proven wrong during implementation.

L8

Execute → KB

Implementation discovers new architectural knowledge for the KB.

L9

Review → Execute

Reviewer finds issues; executor fixes before delivery gate passes.

L10

Deploy → Execute

Release verification fails; delivery sent back for fixes.

L11

Monitor → Interview

Production finding triggers a new Work via a new interview.

Critical: tests must pin HOME

The migration scanner defaults its root to $HOME. Any test that fires it must export HOME=<throwaway> (not just AID_HOME) and include an escape canary, or it will migrate the developer's real repos. This has caused issues 3 times during development.

Open debt — tech-debt.md

Tech Debt

Severity-classified open technical and methodology debt in AID. Read this before starting work in any area — declared debt items and non-obvious gotchas may change your approach or scope.

H1 — Five install manifests must stay lockstep
Severity: HIGH · Area: Distribution

The five install manifests (npm package.json, PyPI pyproject.toml, install.sh, install.ps1, AidInstallCore) must all be updated together when any installed file set changes. A single missed update silently ships a broken install for one channel. Currently managed by manual discipline; a lockstep guard is needed.

M1 — npm and PyPI channels effectively blocked
Severity: MEDIUM · Area: Distribution

Versions 1.1.0 and 1.1.1 are published on npm and PyPI but the channels have outstanding issues (1.1.0 should be deprecated; stale release notes). Affects discoverability and trust.

M2 — Heavy CI gates run on master only
Severity: MEDIUM · Area: Testing

The canonical test suite (tests/run-all.sh) and docs site build run only on push/PR to master. Feature branches skip them, so a direct-to-master merge can break master in multiple ways. Run locally before merging.

M3 — Stale docs/repository-structure.md
Severity: MEDIUM · Area: Documentation

The docs site's repository-structure.md drifted from the actual repo layout. Out of discovery scope to fix; tracked here to prevent it being used as a reference.

L1 — Dead code in install.sh
Severity: LOW · Area: Distribution

The OVERALL_BLOCKED/exit 5/.aid-new protect-on-diff branch in install.sh is unreachable (aid-install-core eliminated the .aid-new path). Safe to remove.

L2 — --sign flag deferred
Severity: LOW · Area: Security

Code-signing for release bundles is planned but not yet implemented. The release pipeline produces unsigned bundles.

L3 — Legacy install path in tests
Severity: LOW · Area: Testing

Some older test fixtures still reference the legacy pre-1.0.0 install layout. They pass but test the wrong path for new-format repos.

L4 — No coverage % metric
Severity: LOW · Area: Testing

Test coverage is assessed by presence/absence of suites, not line percentage. A coverage metric tool would help identify untested paths.
