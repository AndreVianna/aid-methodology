Methodology + CLI tool

At a Glance

AID ("AI Integrated Development") is a step-by-step way of building software with AI agents — and a small command-line tool that installs that way of working into your project. The AI does the heavy lifting; a human stays in charge and approves every step.

What it does for you

  • Understands an existing project — it reads your whole repository and writes a plain Knowledge Base that explains what the project is and how it works.
  • Turns ideas into shippable software — it walks the work through a fixed sequence of phases: gather requirements, write a spec, plan the work, break it into tasks, build them, and ship.
  • Keeps quality high — every phase is reviewed and graded before the work is allowed to move on, and a separate reviewer always checks the work (the author never grades themselves).
  • Lets a human steer — nothing happens automatically between phases; you confirm each one. AID is a guided pipeline, not a runaway robot.
  • Works with five AI tools — the same method installs into Claude Code, Codex, Cursor, Copilot CLI, and Antigravity from one shared source.

Key vocabulary: Canonical, Profile, Work, Phase, Quality gate — see all 29 terms →

Big decisions: One source of truth, everything else generated, Concerns, not project types, drive the docs, The reviewer always outranks the author — see all the "why" →

Concept Spine

Key Vocabulary

AID uses a handful of everyday words in a very specific way. Read these first and the rest of the project will make sense. Each term is defined as AID uses it — not in general.

Canonical

The single source of truth: one folder (canonical/) that holds all of AID's content. Everything else — the installed copies, the per-tool versions — is generated from it. The rule is absolute: edit here, never edit a generated copy.

Relates to Profile Dogfood byte-identity
Profile

A version of AID's content shaped for one specific AI tool. AID supports five (Claude Code, Codex, Cursor, Copilot CLI, Antigravity). A profile is not a user preference — it is "which AI tool's file layout this copy is built for."

Relates to Canonical Render / generator
Dogfood byte-identity

AID runs itself: this repository's own installed copy is built from its own source, and rebuilding it must produce a byte-for-byte identical result. "Dogfood" means the tool uses its own method; "byte-identity" is the guarantee that a rebuild never drifts.

Also called DBI
Emission manifest

A safety list that records every file AID's generator created for a tool. When rebuilding, the generator may only delete files on that list — so it can never accidentally remove your own work. It is the boundary that makes regeneration safe.

Relates to Profile Canonical
Spine

A backbone you walk to check that nothing is missing. AID uses it two ways: the dimension spine is the fixed set of universal questions any project must answer about itself; the concept spine is this glossary's list of load-bearing terms.

Also called concept spine dimension spine
Concern model

The idea that a project's documentation set is decided by the questions a newcomer needs answered (eleven universal "concerns"), not by what type of project it is. This is what lets AID document any kind of work, not just one project shape.

Relates to Spine Doc-set
Doc-set

The confirmed list of Knowledge Base documents for a project. AID proposes a set, you confirm it, and it is recorded in settings. The doc-set is the authority on which documents belong — this very page has one section per doc in it.

Relates to Concern model Spine
Skill

A step-by-step instruction sheet (written in plain Markdown, not code) that an AI tool loads as a slash-command like /aid-discover. Each skill is a small state machine: it knows which step it is on and which AI specialist to hand the work to.

Relates to Agent Phase
Work

The biggest unit AID tracks: one piece of work that travels all the way from requirements to shipped, kept in its own folder with a status file. A work is the parent that contains deliveries, which contain tasks.

Relates to Delivery Task
Delivery

A middle-sized chunk between a work and a task: a working, shippable slice that builds on the one before it. Each delivery has its own lifecycle and its own quality checkpoint before it counts as done.

Relates to Work Delivery gate
Task

The smallest reviewable unit of work. Each task carries a type — such as RESEARCH, DESIGN, IMPLEMENT, or TEST — and the type decides how it is done and how it is reviewed. There are eight task types in total.

Relates to Delivery Execution graph
Triage

An early decision that routes a piece of work down either the full path (every phase, fully gated) or the faster lite path (a single condensed flow for small work). A lite work can be promoted to full if it turns out to be bigger than expected.

Also called lite path full path
Recipe

A pre-filled template for a common small job (add-X, change-X, fix-X). Instead of answering a full interview, you pick a recipe and fill in the blanks. The catalog ships 51 recipes; a recipe just seeds an ordinary lite work, nothing more.

Relates to Triage Work
Phase

One numbered stage of the main pipeline: Discover, Interview, Specify, Plan, Detail, Execute. Each phase is owned by one skill and is gated — a human approves it before the next begins, so the pipeline never advances on its own.

Relates to Skill Quality gate
Render / generator

"Render" is the act of producing a tool-specific copy from the canonical source; the "generator" is the maintainer-only program that does it. It copies, applies one small translation, and is byte-identical on a rerun. It is never shipped to end users.

Relates to Canonical Profile
AID_HOME

The setting that tells the aid tool where to write its changing state (which repos it knows about, install records). It is deliberately kept separate from the read-only folder that holds the tool's own code, which fixed an early install bug.

Relates to Dashboard Bare-box ethos
Human-in-the-middle gating

The governing rule that nothing moves to the next phase without a person approving it. The AI does the work cheaply and quickly; the human sets direction and signs off at each gate. It is the opposite of an autonomous agent left to run on its own.

Relates to Phase Quality gate
Reviewer-tier invariant

A hard quality rule: the agent that grades a piece of work is always at least as capable as the agent that produced it, and the author never grades their own work. Review is a separate, adversarial pass — that is how AID catches its own mistakes.

Relates to Agent Quality gate
Formal feedback loop

One of eleven defined ways a later phase can go back and fix an earlier assumption when reality disagrees. The rule: a loop always leaves a written record — never a quiet workaround — and going back to Discovery is always targeted, never a full redo.

Relates to Phase Knowledge Base
Knowledge Base as gravitational center

The principle that the Knowledge Base is the one shared understanding everything orbits. Every phase reads it, and any phase may update it. The durable truth about a project lives in the Knowledge Base — not in the code, not in the spec.

Relates to Formal feedback loop Doc-set
STATE.md (the tracking spine)

The status file that is the single source of truth for where a piece of work stands. A skill figures out which step it is on by reading this file (the state is in the file, not in the AI's memory). A work with no STATE.md is invisible to the tools.

Relates to Work Dashboard
Content-isolation cornerstone

The cornerstone rule that everything AID installs is clearly labelled and kept separate from your own files (an aid- name prefix, a tracked install list, and a marked boundary in shared files). This is what lets AID install, update, and remove itself cleanly.

Relates to Canonical Emission manifest
Bare-box / dependency-free ethos

A design constraint: AID must install and run on a plain machine with no special packages assumed, and its tools stay dependency-free and predictable. (This is why a database-backed "smart search" idea was deliberately rejected — the Knowledge Base index is a plain table.)

Relates to Dashboard Render / generator
Clean-context review discipline

The rule that a reviewer judges the work "as if a stranger wrote it" — with no knowledge of who produced it or how. The two keystone checks may read only the Knowledge Base, which tests whether the documents alone are good enough to act on.

Relates to Quality gate Reviewer-tier invariant
Agent

One of nine specialist AI personas a skill hands work to — for example the interviewer, the architect, the developer, the reviewer. Each agent is pinned to a capability tier (Large, Medium, or Small). An agent here is a defined role, never a generic free-roaming bot.

Relates to Skill Reviewer-tier invariant
Quality gate

A review checkpoint a phase must pass before it advances. A separate reviewer writes findings into a table, a small program turns that table into a grade from A+ to F, and the phase loops between review and fixing until the grade meets the required minimum.

Relates to Phase Reviewer-tier invariant
Execution graph (and wave)

A map of which tasks depend on which, so independent tasks can run at the same time. A "wave" is one batch of tasks whose dependencies are all met and which may run together — the parallel layers between sequential steps.

Relates to Task Delivery
Dashboard

A read-only local web page that shows the status of every AID project on your machine. It only reads status files and never changes anything; it binds to your own computer only (127.0.0.1) and never reaches the public internet.

Relates to STATE.md Work
Calibration Log

A record, kept per AI dispatch, of how long a task was predicted to take versus how long it actually took. It exists so the time estimates can be tuned from real observation rather than guessed.

Relates to STATE.md Agent

Architecture & Design Rationale

Key Decisions

The significant choices that shaped AID — what was decided, what was rejected, and why. These are the "whys" a newcomer cannot recover from the code alone.

D1 One source of truth, everything else generated Accepted
Decision Write all of AID's content once in canonical/ and generate the five per-tool copies from it with a simple copy-based build, where each copy's deletable footprint is bounded by a safety list.
Rejected
Hand-maintain each tool's copy — five copies inevitably drift apart.
Delete anything not currently generated — unsafe; could destroy a user's own files.
Why AID supports five host tools and the method must stay identical across all of them.
D2 Deterministic, dependency-free tooling Accepted
Decision Keep the generator, the dashboard, and the Knowledge Base index free of third-party dependencies and predictable: standard-library code only, and the index is a plain, diff-friendly table.
Rejected
A database-backed "smart search" over the docs — adds heavy dependencies and unpredictable results that can silently drop a relevant document.
Why AID must install and run on a bare machine with no package-manager assumptions, and its output must be reproducible and reviewable.
D3 Concerns, not project types, drive the documents Accepted
Decision Decide which documents a project needs from the universal questions a newcomer must answer (a fixed eleven-dimension spine), then confirm the set with a human — rather than listing documents per project type.
Rejected
A document catalog per project type (web-app / library / CLI / ...) — you cannot enumerate every type, and every hybrid needs a new entry.
Fix nothing — under-fitting forgets concerns and bundles unrelated things together.
Why AID helps with any kind of digital work, so the derivation must generalize beyond software.
D4 Work out install scope at runtime; no install-time markers Accepted
Decision Decide whether the tool is installed globally or per-user by checking at runtime whether its own folder is writable, and keep the read-only code folder separate from the changing-state folder. Learn about repos as they are encountered — never scan the whole machine.
Rejected
An install-time marker file recording the scope — it goes stale and must be kept in sync.
A machine-wide scan to find AID repos — it scanned the wrong place and caused a real install bug.
Why The tool must work both globally and per-user, and update itself coherently, without a stale marker to maintain.
D5 Twin implementations for cross-platform support Accepted
Decision Maintain the installer in both Bash and PowerShell, and the dashboard in both Python and Node, kept in lockstep by a parity test that checks the two stay equivalent.
Rejected
A single runtime plus a shim — AID targets a bare Windows box (older PowerShell, no guaranteed Python) as a first-class platform, so it cannot assume one interpreter.
Why The "works on a fresh Windows box" promise and the bare-machine bootstrap requirement.
Consequence Shipped scripts must stay plain-ASCII and compatible with the older PowerShell — a rule enforced by automated checks.
D6 Waterfall-with-AI, plus formal feedback loops Accepted
Decision Follow the classic phase order (Understand → Specify → Plan → Build → Verify → Ship), gated by human approval at every step, and add eleven defined feedback loops so any phase can revise an earlier one traceably.
Rejected
Pure Agile / skip to coding — Agile dropped the rigor of discovery and specification, which AID restores now that AI makes building cheap.
Pure waterfall with no loops — real engineering breaks assumptions, so each loop leaves a record rather than a silent workaround.
Why AI makes each phase cheap to iterate, so the bottleneck shifts from "humans are slow" to "humans set direction."
D7 The reviewer always outranks the author Accepted
Decision The agent that grades work is always at a capability tier at least as high as the one that produced it, and the author never grades their own work. Review is a separate pass that loops until the grade meets the minimum.
Rejected
Self-review by the agent that did the work — it misses exactly the oversights the method exists to catch.
Why AID is sold to AI-skeptical adopters who treat any unreviewed slip as sloppy; a separate adversarial review is the guard.
D8 Collapse five renderers into one copy-based generator Accepted
Decision Replace the old five-renderer build with a single generator that copies the source verbatim and applies exactly one surviving translation (renaming for the target tool and resolving which model each agent uses).
Rejected
Keeping a separate renderer per content type — far more transforming machinery than the content actually needed.
Why Keep the generator minimal and predictable while preserving the byte-identical-rebuild guarantee.

Source: decisions.md — 8 decisions (D1–D8). Decisions D1, D2, D4, D5, and D7 are flagged "still load-bearing" (expensive to reverse). Full rationale in the source document.

How the software is built

Architecture

AID is a methodology shipped as tooling: the method is written once in plain Markdown and shell, then compiled into five per-tool install copies. There is one source of truth, and everything else is generated, vendored, or installed from it.

The software has four parts that never share running code — they hand off through committed files instead, which keeps each part simple and independently testable:

Subsystem 1

Render pipeline

A standard-library Python program that turns the single source of truth into the five per-tool install copies, copying verbatim with one small translation step.

Subsystem 2

Install / CLI engine

The aid command, written twice (Bash and PowerShell) so it runs on any machine. It adds, updates, and removes AID inside a repository.

Subsystem 3

Dashboard

A read-only local web view of every project's status. Built twice (Python and Node), it only reads status files, never writes, and binds to your own computer only.

Subsystem 4

Docs site

The public documentation website, built with Astro and Starlight (TypeScript), published to GitHub Pages.

Figure 2. From source to running project: edit the source, generate the five tool copies, publish to three channels, install into a repo, then watch progress on the dashboard. Source: architecture.md

The safety boundary

Each generated copy carries an emission manifest — a list of every file the generator created. On a rebuild, the only files the generator may delete are the ones on that list, so it can never touch your own work. Rebuilding unchanged input is guaranteed byte-for-byte identical.

How the method works

How AID Works

AID is a linear pipeline with formal feedback loops, run human-in-the-middle. Work moves through phases in order; a human approves each one; and when reality breaks an assumption, a defined loop sends the work back to fix the earlier step.

The thirteen user-facing skills fall into five groups; six of them are the mandatory, numbered phases that run in strict order. Brownfield projects start at Discover; greenfield projects skip it and start at Interview.

Figure 3. The work branches once, at Triage. The full path (solid) runs Specify, Plan, and Detail before Execute; the lite path (dashed teal) skips straight to Execute. Source: process-architecture.md

The nine specialist agents, in three tiers

Each skill hands its work to one of nine AI personas. Every persona is pinned to a capability tier — and the reviewer's tier is always at least as high as the author's, so work is never graded by its own author.

Large interviewer architect researcher reviewer
Medium developer operator orchestrator tech-writer
Small clerk

Figure 4. The nine agents across three capability tiers. A profile maps each tier to a concrete AI model. Source: process-architecture.md

Eleven feedback loops

Any phase can revise an earlier one when an assumption breaks — eleven defined loops (L1–L11). For example, Execute can loop back to Discover, Specify, or Detail; Monitor routes a production bug back into Interview. Every loop leaves a written record — never a silent workaround.

How the stages hand off

Pipeline Contracts

The pipeline of skills is held together by four explicit hand-off rules. The most important one: a skill never passes a result to the next skill by message — it writes the result into a status file, and the next skill reads it back.

Contract A

STATE.md hand-off

Each piece of work has one status file that is the single trackable artifact. A spec and tasks alone are not trackable — the status file is the spine.

Contract B

Source → copies (byte-identical)

Regenerating the tool copies from unchanged source is guaranteed byte-for-byte identical; a release gate fails if a rebuild differs from what is committed.

Contract C

State-machine chaining

A single skill run drives its steps to the next natural pause point using exactly four advance types: chain, pause-for-action, pause-for-decision, halt.

Contract D

One coordinated writer

All status-file writes go through one helper with six write modes, using a lock so the parallel task pool never races.

Closed enums keep status readable

The status block uses fixed value lists — a work's lifecycle is one of Running, Paused-Awaiting-Input, Blocked, Completed, or Canceled; a delivery is one of Pending-Spec, Specified, Executing, Gated, Done, or Blocked. These blocks are only ever written by the coordinated writer, never hand-edited.

Who runs what, and when

Workflow Map

AID's workflow is a sequence of skills, each owning a phase, each gated by a human approval before the next begins. Inside each skill, every step hands its work to one of the nine AI specialists.

The thirteen skills fall into five groups; only six are the mandatory, numbered pipeline. The pipeline branches exactly once — at the opening "triage" step of every interview — and closes the loop when Monitor routes a production bug back into Interview.

GroupSkillsRole
Prepareconfig, discoverSet up settings; build the Knowledge Base (brownfield).
Defineinterview, specifyCapture requirements; design each feature.
Mapplan, detailSequence deliveries; break them into typed tasks.
ExecuteexecuteBuild the tasks, with a review loop, in a parallel pool.
Deliverdeploy, monitorPackage and ship; observe production and route findings.
Off-pipelinesummarize, housekeep, query-kb, update-kbOptional helpers, run any time.

The one branch: full vs lite

At triage, the work goes one of two ways. The full path runs Specify, Plan, and Detail before Execute. The lite path collapses those three into one condensed flow and hands straight to Execute — and it is the default for most individual tasks, not a fallback. A lite work can be escalated to full at any point.

Two kinds of parallelism appear: within Execute, independent tasks run together in a coordinated pool; within Discover, the documentation work is split across five research lanes (scout, architecture, analyst, integrator, quality) and then reconciled.

How AID writes its own code

Coding Standards

AID's own conventions for shell, PowerShell, Python, and JavaScript — plus the cornerstone rule that keeps everything AID installs cleanly separated from your own files.

The content-isolation cornerstone

The single most important rule: all AID-delivered content must be namespaced and kept isolated from your content. This is what lets AID install, update, and cleanly remove itself without ever clobbering your work. A non-namespaced AID file is treated as a category violation.

Surface 1

Names

Skills and agents carry an aid- prefix, so they never collide with yours.

Surface 2

Tracked files

Scripts and templates are recorded in a per-repo install list, so removal is precise.

Surface 3

Shared files

Root files like CLAUDE.md are edited in place between marked begin/end boundaries — never with a shadow file.

The cross-platform discipline

Shipped PowerShell must run on a bare Windows box (the older Windows PowerShell, not the newer one), so a list of modern-only constructs is banned and an automated lint catches them. There are two distinct Python floors: the installer targets an older Python for reach, while the dashboard reader needs a newer one. The Python reader is the canonical twin; the Node version mirrors it — so add new behaviour to Python first.

How AID writes its own documents

Authoring Conventions

The rules for AID's own documents, skills, and templates — separate from the rules for its source code. The headline standard: every Knowledge Base document must serve a junior human and an AI agent at the same time.

There are ten authoring principles (P1 through P10). The last and most far-reaching is the dual-audience standard: write one concern per document, write for a junior professional, and use no diagrams in the Knowledge Base itself — with one deliberate exception, the visual summary you are reading now.

Every document has the same shape

In order: a frontmatter block, the title, a contents list (when there are more than three sections), the content sections, and a change log — which is always last. A few fields are required in the frontmatter, and a mechanical check enforces that they are present and parseable.

Skills follow a "thin router" shape: the skill file routes between steps and defers the detail to separate reference files, rather than being one large monolith. Where a rule can be machine-checked it is enforced by a blocking automated check; where it is a matter of prose quality, the reviewer agent is the authority.

How AID is tested

Test Landscape

AID is a hybrid repository with no single test runner. It tests six different surfaces, each with the right tool for the job, all driven from one entry point.

The bulk of the testing is a large set of shell test suites — the single entry point (tests/run-all.sh) discovers them by pattern and runs each in its own isolated process with a time limit. The Windows tests live separately and run only on a Windows machine, executed twice (under both the new and the old PowerShell) because the product advertises support for the older one.

SurfaceHow it is tested
Installer / CLIThe large shell suite (the bulk of coverage) plus a Bash↔PowerShell parity suite.
Windows installerA separate PowerShell suite, run twice on a Windows machine (new and old PowerShell).
Dashboard readerPython tests plus matching Node tests, kept in parity.
Docs siteUnit tests, plus the full site build as the heavier gate.
Profile generatorBuilt-in self-test suites; five run in the automated checks.

No silent green

The automated checks assert that the required runtimes are actually present, so a test sub-suite can never silently skip and report a false pass. There is no coverage percentage gate — coverage is judged by which suites exist, not by a number.

What to know before you change things

Known Tech Debt

An honest diagnosis of the project's open weak spots and change-time gotchas — so work doesn't add more of them. The good news first: the codebase is healthier than its size suggests, with zero "to-do / fix-me / hack" markers in the installer code.

Six debt items are tracked. The two biggest themes both come from the same deliberate choice — keeping twin implementations — which means some behaviour must be written and changed twice.

TD-001 · Dual-language duplication High

The install engine (Bash + PowerShell) and the dashboard reader (Python + Node) are each maintained twice. A behaviour change must be made in both, and a parity test guards each pair.

TD-002 · Large engine files Medium

A few files are very large (the dashboard reader, the Windows CLI, the install engine), which raises the effort to change them safely.

TD-003 · Windows lane sits outside the main suite High

The Windows tests are not part of the single test entry point, so a green local run does not prove the Windows behaviour.

TD-004 · Heavy gates run on the main branch only Medium

Some correctness checks only run when merging to the main branch, so feature branches can look green while still missing a gate.

TD-005 · A flaky parity test case Low

A handful of parity test cases were unreliable and have been dropped.

TD-006 · Prompt-driven skills are hard to test mechanically Medium

The skills are written as prose state machines, which are harder to cover with automated tests than code.

Healthy on security

No secrets in the repository; automated-check actions are pinned to exact versions and bumped weekly; publishing uses token-less, attested release; and no vulnerability-bearing runtime dependencies were found (the packages have zero runtime dependencies).

Change-time gotchas

Edit the source of truth, never a generated copy. Pin the home directory in any test that triggers the migration scan. Keep shipped scripts plain-ASCII. The main branch is protected and publishes are irreversible. Resolved debt items are deleted entirely — git history is the only audit trail.

From source to shipped

Infrastructure

AID is a methodology shipped as tooling, not a hosted service — there is no production server, container, or database to operate. Its "deployment" is publishing an installer to three channels and a docs site to GitHub Pages.

The code lives on GitHub with a protected main branch (changes go through a pull request with passing checks). Four automated workflows run on changes, all pinned to exact versions and bumped weekly.

Channel 1

GitHub Release

Five per-tool archives plus a checksum file. Always published.

Channel 2

npm

Token-less, attested publish; pre-checks and skips if the version already exists.

Channel 3

PyPI

Token-less, attested publish via trusted publishing; skips existing versions.

The release sequence

Tag a version → one gate job re-checks version-sync, rebuild-drift, and the full test suite → the GitHub release runs, then npm and PyPI publish in parallel. All three carry the same single version, kept in sync by an automated check.

The dashboard runs locally in either Node or Python, binding to your own computer by default. There is an optional secure remote mode that exposes it only to your own authorized devices over an encrypted private network — never the public internet.

How AID decides "good enough"

Quality Gates

How AID decides whether a phase's work is good enough to advance. This is a three-part loop, not a single test: a reviewer finds issues, a small program turns them into a grade, and the work loops until the grade meets the minimum.

Figure 7. The quality gate loop. If the grade meets the minimum the phase advances; otherwise the work loops back to fix and re-review (dashed). Source: quality-gates.md

The grade comes from counting findings by severity: the worst severity present sets the grade, and the count within it nudges the modifier. An empty findings list is an automatic top grade. The required minimum is configurable, defaults to A, and several phases require the strictest setting (zero open findings of any kind).

Two keystone checks

The Knowledge Base review uses a four-mandate panel. Two of them are keystone hard gates: "teach-back" (could a stranger learn the project from the docs alone?) and "act-back" (could they act on it?). An open failure on either blocks discovery from closing — even if the numeric grade would otherwise pass. Both may read only the Knowledge Base, which tests whether the docs alone are sufficient.

Alongside the human review, a set of mechanical checks gate automatically: rebuild-drift, a "no ungrounded terms" closure check, freshness checks, and the generator self-tests. The release is blocked behind these — publishing cannot happen until they pass.
