## Summary

While authoring task-033's REAL-writer round-trip tests, `write-control-signal.sh`
was found to create the `task.stop`/`task.resume` control-signal file at a
**garbled, wrong path on Windows** — HTTP 200 is still returned (the writer's own
exit code never reflects the mistake), but `stop_requested` silently never
flips. This is a genuine, previously-unexercised production defect in
FR-T3 (Task Stop/Resume), not a task-033 test-authoring issue. **This does NOT
block task-033** — the affected assertions are `skipIf`-gated (fully exercised
and passing on POSIX/CI) and a regression canary documents the current
behavior; task-033 itself is otherwise complete. This impediment exists purely
to route the production fix.

## What Was Found

**Expected (from task-028/029 DETAIL.md / SPEC.md "`write-control-signal.sh`
contract"):**
```
--action stop: mkdir -p the control dir; atomically create
<work_dir>/../../.control/<work_id>/task-<NNN>.stop
```
where `<work_dir>` is `AID_WORK_DIR`, the dispatcher's own `resolve_work_dir()`
output.

**Actual (in the codebase):**
`_op_task_stop_argv`/`_op_task_resume_argv` (both server.py and server.mjs) set
`AID_WORK_DIR` via a plain `str(work_dir)` / `String(workDir)` — the native
path form. On Windows this is **backslash-separated**
(`C:\Users\...\work-702-stopresume`). `write-control-signal.sh`'s own path
arithmetic assumes a **forward-slash** path:

```bash
WORK_DIR="${AID_WORK_DIR:-$(pwd)}"
WORK_DIR="${WORK_DIR%/}"          # no-op: no trailing forward slash to strip
WORK_ID="${WORK_DIR##*/}"          # BUG: no '/' anywhere in a backslash path,
                                    # so this returns the WHOLE path unchanged
CONTROL_DIR="${WORK_DIR}/../../.control/${WORK_ID}"
```

`WORK_ID` ends up being the *entire* backslash `AID_WORK_DIR` value again
(never just the basename), so `CONTROL_DIR` embeds the whole Windows path a
SECOND time as a bogus nested path component — the signal file is created at
a garbled location instead of
`<work_dir>/../../.control/<work_id>/task-<NNN>.stop`. The script still exits
`0` (its own arithmetic never errors — it just computes the wrong path), so
the dashboard returns HTTP 200 to the client, and the reader (which derives
the same directory via `pathlib`, slash-agnostic) looks at the CORRECT path
and finds nothing — `stop_requested` never flips. `pipeline.finish`
(`writeback-state.sh`) is **unaffected**: it never does string-arithmetic on
`$AID_WORK_DIR`/`$AID_STATE_FILE`, only uses them as opaque file paths for
redirection (real filesystem I/O accepts either slash style on Windows).

**Evidence** (reproduced twice, standalone script AND through the real
dispatcher; both server.py and server.mjs affected identically — see
`dashboard/server/tests/test_task033_execution_control_round_trips.py`'s
`TestKnownDefectAidWorkDirBackslashOnWindows` regression canary, which
currently PASSES by asserting today's buggy behavior):

```
$ python - <<'EOF'
... AID_WORK_DIR="C:\Users\...\aid-repro-usqeiils\.aid\works\work-999-repro"
... invoke write-control-signal.sh --task-id 001 --action stop
EOF
returncode 0
stdout OK: C:\Users\...\work-999-repro/../../.control/C:\Users\...\work-999-repro/task-001.stop created -- stop signal for task-001
expected signal exists: False
tmp tree under .aid:
  C:\Users\...\aid-repro-usqeiils\.aid\.control
  C:\Users\...\aid-repro-usqeiils\.aid\.control\C\Users\...   (colon PUA-escaped by MSYS; nested nonsense tree)
```

The `_dispatch_op`-level reproduction (Python) and the Node twin
(`dispatchOp` via a sliced import) are BYTE-IDENTICAL to each other (both
report HTTP 200 with the `{"ok":true,"op":"task.stop"}` envelope) — the two
runtimes are broken in exactly the same way, so cross-twin dispatch-response
parity (AC4) is unaffected by this defect; only the WRITER's actual
filesystem effect is wrong.

## Options

### Option A: posix-ify the env value at the argv-builder call site (recommended)
**Approach:** In `_op_task_stop_argv`/`_op_task_resume_argv` (server.py) set
`env = {"AID_WORK_DIR": work_dir.as_posix()}` instead of `str(work_dir)`;
mirror in `opTaskStopArgv`/`opTaskResumeArgv` (server.mjs) via
`workDir.split(sep).join("/")` (or equivalent) instead of `String(workDir)`.
This is the SAME convention `_posix_argv_path()`/`.as_posix()` already
applies elsewhere in server.py for ARGV path elements reaching a bash child
(see `_run_writer`'s own docstring) — just never applied to this ENV VAR.
**Effort:** S
**Risk:** Low — a pure string-representation change; no behavior change on
POSIX (`.as_posix()` is a no-op there).
**Scope impact:** Touches only the two argv-builder functions (both twins) —
no OP_TABLE shape change, no new endpoint, no schema_version bump.
**Spec impact:** None (SPEC.md's contract text is already correct; this
brings the implementation in line with it).

### Option B: make write-control-signal.sh itself path-separator-agnostic
**Approach:** Rewrite the script's own path arithmetic to normalize
`AID_WORK_DIR` (e.g. `WORK_DIR="${WORK_DIR//\\//}"` before the existing
`${WORK_DIR##*/}` logic) so it tolerates a backslash-separated input
regardless of what the caller passes.
**Effort:** S
**Risk:** Low-Medium — touches the writer script itself (co-vendored to 5
profiles + `dashboard/scripts/`); needs re-verification against
`tests/canonical/test-write-control-signal.sh`.
**Scope impact:** Slightly larger (co-vendored script + MANIFEST-tracked
copies) than Option A.
**Spec impact:** None.

### Option C: defensive belt-and-suspenders — do both A and B
**Approach:** Apply Option A (correct the root cause at the call site) AND
Option B (make the writer itself defensively tolerant), so a FUTURE caller
that reintroduces a native-path `AID_WORK_DIR` doesn't silently regress.
**Effort:** M
**Risk:** Low.

---

## Recommendation

Option A — it is the minimal, precedented fix (mirrors an existing convention
in the same file), touches only the two Python/Node argv-builder functions,
and requires no writer-script re-verification. The user/Architect decides
whether to also apply Option B's defense-in-depth.
