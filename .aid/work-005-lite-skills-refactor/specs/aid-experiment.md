## 1. Why this one is different

`aid-experiment` was initially binned as a "clear mismatch," but that was the wrong cut.
For review/research/report there is nothing costly or irreversible to gate before the
"doing" ‚Äî the deliverable *is* the analysis, so planning-then-halting just withholds it.
An experiment is the opposite:

- the **hypothesis + pre-registered, measurable acceptance criteria** must be fixed
  *before* running (defining success after seeing results invalidates the experiment);
- **running is the costly / often irreversible step** (especially a live A/B on real
  users), so the design must be human-approved first.

So the plan ‚Üí approve ‚Üí run+analyze cycle is a **feature**, not red-tape. `aid-experiment`
therefore **keeps the cycle** and is reclassified out of the collapse set. What was
lacking is not the machinery but the **rigor of the content** the skill authors into the
standard documents ‚Äî which is exactly the skill's own objective.

## 2. What stays (unchanged)

Everything structural:

- **Engine-driven** ‚Äî thin generated doorway ‚Üí shared shortcut engine ‚Üí halt before
  Execute; `/aid-execute` runs it. `aid-experiment` is **not** `repurpose: true`; its
  `SKILL.md` stays generated by `build-shortcut-skills.py`.
- The **standard document set** (REQUIREMENTS / SPEC / PLAN / BLUEPRINT / DETAIL).
- The **normal GATE** (grades those documents ‚Äî *no* special-validation gate mechanism;
  the validation is expressed as document content and graded normally).
- The **DETAIL task chain** `RESEARCH ‚Üí [IMPLEMENT] ‚Üí RESEARCH` (design ‚Üí build variants
  if needed ‚Üí run + analyze against the significance criteria ‚Üí recommendation).
- Run flavor (offline vs. live) is decided at Execute, not here; live/field runs are a
  deploy + collect-over-time step, with analysis in a later run ‚Äî outside this planning
  skill.

The scaffolding authors the captured rigor *into* REQUIREMENTS (this is where "the rigor

  bounds **+** the sample-size/power precondition **+** the duration/stopping rule.

### 3.3 Validation ‚Üí SPEC

Give `aid-experiment` a **defined experimental-design section in SPEC** (today it is "base
sections only"). It specifies *how* the validation is performed ‚Äî the "special validation‚Ä¶
defined in the spec":

- independent / dependent variables;
- control vs. treatment assignment + randomization;
- metric computation (primary + guardrails);
- sample-size / power calculation;
- the statistical method / test, and how significance and the stopping rule are evaluated.

Unchanged. The `RESEARCH ‚Üí [IMPLEMENT] ‚Üí RESEARCH` chain, the normal gate, and Execute
