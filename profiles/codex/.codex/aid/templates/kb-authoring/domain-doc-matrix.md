# KB Authoring -- Domain -> Doc-Set Matrix

> Curated, structured data artifact mapping a project's **domain** to the KB doc-set it
> should produce. Each row's docs are anchored to the domain-agnostic **dimension spine**
> (see [concern-model.md](concern-model.md)). Consumed by `aid-discover` (doc-set proposal:
> matrix-lookup -> research-on-miss -> propose -> confirm). The **software-cli** /
> **software-web** required rows reproduce today's 15-doc `synth_default_seed` exactly, so
> existing behavior is preserved as one cached row.
>
> This is a `canonical/` artifact: it ships, is installed, and is changed only by
> release / human curation. There is no automatic install -> canonical feedback
> (feature-014 / FR-40). See [README.md](README.md) for the full kb-authoring index.

## What this artifact is

`/aid-discover` derives a project's KB doc-set from its **domain**. AID helps with *any*
kind of digital work -- software, data/ML, content, research, design, ops,
methodology/tooling -- and each domain documents itself differently. A fixed software
taxonomy is wrong for non-software work (it forces empty buckets) and incomplete for
software work (it misses project-specific concerns).

This matrix is the **fast path**: a curated table of `domain -> doc-set`. On a **hit**, the
domain's row resolves to a doc-set directly. On a **miss** (a novel or hybrid domain not
listed here), discovery falls back to **researching** the domain's documentation practices
and synthesizing a set (flagged `auto-researched`). Either way the result is **anchored to
the single fixed spine**, **composable** for hybrids, and **proposed -> confirmed** with the
user before it is persisted to `.aid/settings.yml -> discovery.doc_set`.

The matrix never replaces the spine. The spine answers "did we cover everything?"; the
matrix answers "what is the conventional doc-set for this domain?". A matrix row is a
**curated convenience**, not a new source of truth -- every doc in every row maps to exactly
one spine dimension.

## Schema

Each matrix **row** is a domain. Each domain owns a list of **doc records**. A doc record is
four fields:

| Field | Meaning | Allowed values |
|-------|---------|----------------|
| **filename** | Basename under `.aid/knowledge/`. Joins to the doc's frontmatter (`kb-category`) and to `document-expectations.md` (keyed by `### <filename>`). | any `*.md` basename, no commas, no pipes |
| **spine-dimension** | Which fixed spine dimension this doc realizes. The anchor that makes "did we cover everything?" answerable. | `C0`..`C9`, `D`, or `meta` (orientation, not a newcomer concern) |
| **owner** | The freshness-accountable role (an `aid-researcher` slot or `skill-self`). Same enum as `discovery.doc_set` field 2. | `aid-researcher-scout` \| `aid-researcher-architecture` \| `aid-researcher-analyst` \| `aid-researcher-integrator` \| `aid-researcher-quality` \| `skill-self` |
| **presence** | Whether the doc is always produced for this domain, or only when a condition holds. | `required` \| `conditional[:<when>]` |

Each **domain row** also carries a row-level **provenance** field (not per-doc):

| Row field | Meaning | Allowed values |
|-----------|---------|----------------|
| **provenance** | How this row entered the matrix. | `curated` (hand-authored + standards-grounded) \| `auto-researched` (synthesized by the research fallback) |

**Notation in the tables below.** Each domain section lists its docs as a table whose
columns are exactly `filename | spine-dimension | owner | presence`. This is the same shape
that lands in `discovery.doc_set` (filename | owner | presence), with the spine-dimension
column added here for authoring traceability. When a row is materialized into
`discovery.doc_set`, the spine-dimension column is dropped (it is documentation, recoverable
from this matrix or `concern-model.md`); `discovery.doc_set` records only
`filename | owner | presence`.

> **Why `owner` reuses the `aid-researcher` enum.** feature-014 introduces no new agent enum
> value. Every doc -- in any domain -- is filled by one of the five parameterized
> `aid-researcher` slots, or generated by the skill itself (`skill-self`). The slot names are
> software-flavored but the roles generalize: *scout* = surface/inventory, *architecture* =
> structure/decisions, *analyst* = parts/conventions/data, *integrator* = connections/
> vocabulary, *quality* = testing/risk/operation. Non-software rows reuse the nearest-fit
> slot; the propose -> confirm gate lets the user re-assign.

## The dimension spine (the anchor)

Every doc in every row maps to one of the **11 fixed spine dimensions** from
[concern-model.md](concern-model.md). The spine is a T2 cardinality contract (exactly 11
dimensions); it does **not** grow per domain. Only the *realization* (which docs, named how)
varies. Restated here for lookup:

| Id | Spine dimension | The universal question |
|----|-----------------|------------------------|
| C9 | What it does for users | What does it do for its users / what are its capabilities? |
| C1 | Build & shape | How is it built? What is its overall structure / anatomy? |
| C2 | Parts & connections | What are the parts and how do they connect? |
| C0 | Technology / medium | What is it built *with*? |
| C3 | Conventions | What conventions and standards does it follow? |
| C4 | Vocabulary | What is its native vocabulary / ubiquitous language? |
| C5 | Data & contracts | What are its data shapes and structural contracts? |
| C6 | Quality & checking | How is it checked? How healthy is it? |
| C7 | Risk & debt | What is risky, owed, or worked around? |
| C8 | Shipping & operation | How does it ship and run? |
| D | Decisions & rationale | Why is it the way it is -- what was decided, why, what was rejected? **(conditional)** |
| meta | Orientation / stakeholders | Cross-cutting registry/completeness docs (`README.md`, `external-sources.md`); stakeholders is answered by the pipeline, not the KB. |

**Coverage invariant (FR-37).** A resolved doc-set is valid only when **every spine
dimension is covered by at least one doc, or explicitly marked conditional**. `D` (Decisions)
is conditional by default. The matrix rows below satisfy this per domain.

## Provenance: curated vs auto-researched

Every matrix row carries a **provenance** field recording how it entered the doc-set:

| Provenance | Meaning | Where it lives | Trust |
|------------|---------|----------------|-------|
| `curated` | Hand-authored in this `canonical/` matrix, standards-grounded, reviewed at release. The eight rows above are all `curated`. | `domain-doc-matrix.md` (this file) | Deterministic per release; the fast path. |
| `auto-researched` | Synthesized by the research fallback when the classifier hits a domain **not** in this matrix (novel or hybrid). Discovery researches the domain's documentation practices, maps each proposed doc to a spine dimension, and flags the result. | The project's `.aid/settings.yml -> discovery.doc_set` (persisted locally) and, optionally, an emitted `.aid/generated/domain-doc-candidate.md` for upstream PR. | Proposed -> confirmed with the user; never auto-final. |

**The promotion path (dogfood loop, closed by hand).** An `auto-researched` row proven useful
on a real project may be **curated** into this matrix by a maintainer via a normal commit
(optionally seeded by the project's `domain-doc-candidate.md`). There is **no automatic
install -> canonical feedback**: the install boundary is one-directional (canonical -> user).
A row's provenance flips from `auto-researched` to `curated` only by that human curation step.

**Both provenances anchor to the same spine.** A research-synthesized row is held to the same
coverage invariant as a curated one: every spine dimension covered by >=1 doc or explicitly
conditional. Research changes *where the doc-set comes from*, never *what it must satisfy*.

## Hybrid composition rule

Real projects are often hybrids -- a data-ml project with a web dashboard, a methodology
project that ships as a CLI. Discovery resolves a hybrid by **composing** the relevant domain
rows, **never** by picking one bucket or by treating domains as mutually exclusive.

The rule, in order:

1. **Identify the relevant rows.** The domain classifier may confirm more than one domain
   (e.g. `data-ml` + `software-web`). Take each confirmed domain's row.
2. **Union the docs over the single spine.** Lay all docs from all rows onto the one fixed
   11-dimension spine. The spine is shared; the rows are not separate taxonomies, they are
   different *realizations* of the same spine.
3. **Dedupe by spine-dimension, not by filename.** When two rows both realize the same
   dimension (e.g. `data-ml`'s `data-schemas.md` and `software-web`'s `schemas.md` both
   realize C5), do **not** keep both blindly. Propose the better-fit doc for that dimension,
   or -- when the project genuinely needs both views -- a deliberate split under the
   dimension (the three-force boundary rule in [concern-model.md](concern-model.md)). The
   user confirms which docs realize each dimension at the propose -> confirm gate.
4. **Promote presence on conflict.** If the same dimension is `required` in one row and
   `conditional` in another, the composed result is `required` (the stronger commitment).
5. **Walk the spine to verify coverage.** After composition, confirm every spine dimension is
   covered by >=1 doc or explicitly conditional. This is the same coverage invariant; the
   spine -- not any single domain row -- is what answers "did we cover everything?".

**Never mutually-exclusive buckets.** A hybrid is not "70% data-ml, 30% web -> pick data-ml".
It is the *union* of both rows over the shared spine, deduped by dimension. There is exactly
one spine; the domain rows are overlays on it, and overlays compose.
